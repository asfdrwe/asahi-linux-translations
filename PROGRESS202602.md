# 進捗報告：Linux 6.19

遅くなりましたが、あけましておめでとうございます！  
Linux 6.19がリリースされました……はい、早速本題に入ります。皆さんが知りたいことはわかっています。

## 大ニュース

Asahi Linux は今年で5周年を迎えます。この5年間で、シリアルポート経由の『Hello World』から、Linux エコシステムで最高レベルのサポートを受けた AArch64 デスクトップ
品質のプラットフォームの一つへと成長しました。Asahi Linux への継続的な注目が、多くの開発者に AArch64 を本気で取り組むきっかけを与えました。人気ソフトウェアの
プラットフォーム固有バグが次々と修正され、Apple Silicon 上の Linux で使えるようになりました。これまでの成果を非常に誇りに思っており、このプロジェクトは
大成功であり続けています。

それでも、誰もが口にする一つの質問がずっとありました。  
あらゆる発表でも上流への統合成功でもブログ投稿でも必ずと言っていいほど出てくる質問です。IRC や Matrix では週に最低1回は聞かれ、メールでも時々届きます。

『USB-C 経由のディスプレイ出力はいつ対応ですか？』  
『DisplayPort Alt Mode の予定は？』  
『MacBook Air で HDMI アダプタ使えるようになりましたか？』

特定の機能の対応予定時刻を聞かないでくださいと何度も丁寧にお願いしたにも関わらず、質問は止みませんでした。対策として『最低でもこの日付』と設定し、質問が来るたびに
倍にする案も出ましたが、すぐに宇宙の熱的死滅予測日を超えてしまいました。結局、id Software 方式の定番回答に戻りました。

DP Alt Mode　は完了したときに完了です。

そして……完了しました。ある程度ですが。

12月にSven 氏が[39C3で講演](https://media.ccc.de/v/39c3-asahi-linux-porting-linux-to-apple-silicon)を行い、これまでの Asahi の物語、リバース
エンジニアリングの過程、そして近未来について語りました。講演の最後に、スライドが M1 MacBook Air 上で動作しており、USB-C → HDMI アダプタ経由で会場の A Vシステムに
接続されていたことを明かしました！

同時に、[fairydustブランチ](https://github.com/AsahiLinux/linux/tree/fairydust)を下流 Linux ツリーで静かに公開しました。このブランチは Sven氏、Janne氏、marcan氏 に
よる長年の苦労の結晶で、このプラットフォームの脆く複雑なUSB・ディスプレイスタックを何とか制御・飼いならしたものです。Apple SiliconのUSB-Cポートからディスプレイ信号を出すには、
DCP[^1]、DPXBAR[^2]、ATCPHY[^3]、ACE[^4] の 4 つの異なるハードウェアブロックが必要です。それぞれにリバースエンジニアリング、Linux ドライバ作成、そしてお互いが
協調動作するように調整が必要でした。

とはいえ、まだやるべきことはあります。  
現在の fairydust ブランチでは、特定の USB-C ポートだけを DisplayPort 用に『有効化』しているため、複数の USB-C ディスプレイ はまだ使えません。
コールドプラグ・ホットプラグ時の挙動にもクセがあります。また、一部のユーザーが報告しているように、特定のディスプレイ設定で DCP が正しく動作せず、
色が不正確・過飽和になったり、タイミングモードが欠落したりする問題もあります。

こうした理由から、fairydust ブランチは現状のまま提供します。主に開発者がこれらの問題を最小限の対応で解決する手助けになることを目的としています。Apple Silicon　で
自分でカーネルをビルド・インストールできる方はぜひお試しください。ただし、一般向けに準備ができるまではサポートはできません。

## もう一つの大ニュース

かなり前から m1n1 には M3 シリーズへの基本対応が入っていました。足りなかったのは、各機種ごとの DeviceTree と、M3 特有のハードウェア挙動や M2 からの変更に
対応した Linux カーネルパッチでした。既存のパッチセットが扱いやすくなってから本格的に取り組む予定でしたが、基礎作業が新しい貢献者を刺激して手伝ってくれることを
密かに期待していました。結果、*3人*の新しい貢献者が現れました！

[Alyssa Milburn 氏 (noopwafel)](https://noopwafel.net)、[Michael Reeves 氏 (integralpilot)](https://github.com/IntegralPilot)、[Shiz 氏](https://shiz.me)の
3 人が Janne 氏の助けを借りて予備的な DeviceTree を作成し、驚くほど多くのハードウェアがほとんど変更なしで動くことを確認しました。NVMe と割り込みコントローラに軽微なカーネル修正を
加えたところ、Michael 氏が M3 MacBook Air で[KDE Plasmaまでブート](https://www.reddit.com/r/AsahiLinux/comments/1qnddjd/m3_now_has_fedora_43_asahi_remix_working_with_kde/)に成功しました！

つまり現在の M3 対応状況は、最初の Arch Linux ARM ベータを出した頃の M1 対応状況とほぼ同等です（キーボード、タッチパッド、WiFi、NVMe、USB3 が動作。ただし m1n1 と
Asahi カーネルにまだ統合されていないのでローカルパッチが必要）。ではもうすぐリリースですか？そう簡単ではありません。

5 年で状況は大きく変わりました。私たちは『最も完成度が高く洗練された AArch64 デスクトップLinux体験』として、また一般的なデスクトップ Linux としても
*トップクラス*の評価を得ています。この評価はとても誇らしいもので、多くの個人的な尽力の上に成り立っています。無駄にしたり当たり前だと思ったりするつもりはありません。

理想的には、現在の M1 / M2 対応レベルが M3 の一般公開の最低ラインであるべきです。でもそれは現実的でないですし、昔の初期 ALARM のような中途半端なものを出すのも避けたいです。
ではリリースまでに何が必要でしょうか？非常に多くあります。

まず最初に気づくのは、グラフィックが完全ソフトウェアレンダリングであることです。大変遅く、消費電力も大きく、ターミナルでテキストをスクロールするのもやっとです。
残念ながらこれは当面改善しそうにありません。M3 シリーズの GPU は M1 / M2 と大きく異なり、ハードウェアレイトレーシング、メッシュシェーダー、Dynamic Caching などが
追加されています。Alyssa 氏と Michael 氏が M3 GPU のリバースエンジニアリングに取り組み、dougallj 氏や TellowKrinkle 氏の成果を基に M2 から M3 への GPU ISA 変更に
関して多くの進展を見せています。

また、現在は iBoot が DCP を初期化しフレームバッファを提供してもらう形になっており、自分たちで DCP を正しく駆動できていません。これが非常に遅く非効率で、
バックライトなどの多くのディスプレイ機能がまともに制御できません。M3 は macOS 13.5 で動かず、macOS 14 で DCP ファームウェアインターフェースが変わっているため、
さらにリバースエンジニアリングが必要です。幸いプロトコル部分は変わっていないので、既存ツールを少し修正するだけで新しいインターフェースをトレースできます。

ハードウェア有効化以外にも、Asahi にふさわしい多くの仕上げ作業が必要です。Energy-Aware Scheduling、スピーカー保護 ＆ EQ 調整、マイク・ウェブカメラ対応など、
まだない機能がたくさんあるので、しばらくかかります。このうち Energy-Aware Scheduling はリリースを止めない品質管理機能ですが、speakersafetyd での M3 対応などは
リリースブロッカーになっています。

M3 対応が公開可能になるまで*それほど*長くはかからないと期待していますが、他のすべてと同じくETAはお答えできませんし、聞かないでくださいね。

## Hertz so good（120 Hz、めっちゃいい）

14インチおよび16インチのMacBook Proは非常に優れたディスプレイを搭載しています。色再現性が極めて正確で、非常に明るく、**しかも120 Hzのリフレッシュレートに対応**しています。ただし、ひとつ落とし穴があります。

macOSでは、単にディスプレイを120 Hzに設定して終わり、というわけにはいきません。Appleは60 Hzを超えるリフレッシュレートを「ProMotion」という機能の裏に隠しています。これは実質的に普通の可変リフレッシュレート（VRR）のマーケティング用語に過ぎません。macOS特有のクセだと思い込み、DCPファームウェアで120 Hzのタイミングモードを選べばLinuxでもそのレートで駆動できるだろうと考えるのは無理もないですが、**実際にはそうではありません**。

Appleだけが知っている理由で、DCPはsurface swap request構造体の特定の3つのフィールドが埋められていない限り、MacBook Proのパネルを60 Hz超で駆動することを拒否します。これらのフィールドが何らかのタイムスタンプであることは以前からわかっていましたが、それ以上深く調べる時間がありませんでした。そこで、またしても新しい貢献者が登場！

[Oliver Bestmann](https://github.com/oliverbestmann)さんが自らMacBook Proで120 Hzを実現することを目標に、これらの3つのタイムスタンプを調査しました。macOSのトレースを分析したところ、これらはCPUタイマーのティック単位で増加しており、ほぼ常に1フレーム間隔であることがわかりました。これはフレーム提示時刻の管理に使われていることを示唆しています。提示時刻管理はVRRを正しく動作させるために必要で、コンポジタとドライバの両方が、特定のフレームが実際にディスプレイに表示された時刻を知る必要があります。VRRが無効でも、コンポジタはこの情報を使ってフレームペーシングを安定させ、ティアリングを最小限に抑えることができます。

現時点では一貫した120 Hzだけを目標としており、VRRはまだ対象外です。macOSが両者を密接に結びつけているため、DCPが120 Hzのために何を期待しているのかを正確に把握するのは困難です。タイムスタンプが必要なのは明らかですが、なぜ？ DCPは何をするのか？ そして**正確に**何を表しているのか？

時には「バカなこと」をするのが実は賢い選択です。Oliverさんは「タイムスタンプはVRRでのみ意味がある」と仮定し、各タイムスタンプフィールドに**静的な値**を詰め込んでみました。すると……動きました！ kernel 6.18.4以降、14インチ/16インチMacBook Proのオーナーは内蔵ディスプレイを120 Hzで駆動できるようになりました。

もちろん、この解決策は明らかに「jank（ハックっぽい）」です。提示タイムスタンプは現在、KMSサブシステムがatomic state flushをトリガーするたびに設定されており、静的値にするのは本来想定されていません。この方法は私たちのユースケースでは機能しますが、VRRサポートを不可能にします。これが次の話題にぴったりつながります。

## Landing the plane（着陸開始）

Linux向けDCPドライバはこれまでかなり不完全でした。これは驚くことではありません。ディスプレイエンジンは極めて複雑で、それを反映してDCPが動作するファームウェアは**9 MiB**もの巨大なblobです。このファームウェアはmacOSと密接に統合されるよう設計されたインターフェースを公開しており、macOSリリース間で破壊的に変更されるため、バージョン付き構造体や関数呼び出しの特別な扱いが必要です。

これらが積み重なって、ドライバは最適とは言えない、部分的な形で開発されてきました。その理由はたくさんあります：

- 時間的余裕がなかった（特にJanneさんはAsahiカーネルツリーのメンテナンスとリベースの負担を一手に引き受けていました）
- 他のハードウェアの立ち上げなど、もっと優先度の高い作業があった
- いずれにせよファームウェアのバージョン管理をより良く扱うためにRustでドライバを書き直す予定だった

それに加えて、当時の設計目標では単に問題なかったのです。初期目標はラップトップの内蔵ディスプレイとデスクトップのHDMIポートを確実に駆動できる程度のDCPを立ち上げることでした。それを達成するために、DCPのファームウェアインターフェースのごく一部をKMS APIに無理やりくっつけて、各スワップで単一の8ビットARGBフレームバッファをスキャンアウトできるようにしただけです。

その後、DisplayPort/HDMI経由のオーディオサポート、Night Light向けの基本的な色管理（Colour Transformation Matrices対応）、初歩的なハードウェアオーバーレイを実装しました。しかしHDR、VRR、他のフレームバッファフォーマット対応、外部ディスプレイのハードウェア輝度制御（DDC/CI）、マルチメディアやフルスクリーンアプリ向けの直接スキャンアウトなど、まだ多くの機能が残っています。

現在のドライバアーキテクチャの枠内でこれらをサポートするのは困難です。ユーザースペースとの統合やKMS APIとの一部コンポーネントの相互作用に未解決の問題がいくつもあります。それでも新機能を進めたいし、Rust KMSバインディングが上流にマージされるのを待っていたらかなり時間がかかるため、必要に応じて既存DCPドライバの部分的なリファクタリングを開始しました。最初に取りかかったのはハードウェアプレーン処理のコードです。

なぜプレーンから？ ハードウェアプレーンの適切なサポートはパフォーマンスと効率にとって重要です。ほとんどのディスプレイエンジンは複数のフレームバッファをハードウェアで合成する機能を持っており、DCPも例外ではありません。レイヤリング、移動、ブレンド、基本的な色変換が可能です。典型的なユースケースはカーソルです。カーソルが動くたびにGPUがデスクトップ全体を再描画する代わりに、カーソルをディスプレイエンジンのオーバーレイプレーンの1つに置き、静的なフレームバッファを画面上移動させるだけにします。GPUは画面上の**コンテンツ**が再描画を必要とする場合（ボタンにホバーしたときなど）にのみアクティブにレンダリングします。

以前に極めて限定的なサポートを無理やり入れたところ、Plasma 6のハードウェアカーソルサポートと上手く動作していました。しかしもっと深く掘り下げる必要があります。

DCPは非常に便利な機能を持っており、その一部はHDRや直接ビデオスキャンアウトに**絶対必要**です。特に重要なのは：

- セミプラナーY’CbCrフレームバッファ（SDRおよびHDR）の直接スキャンアウト
- 異なる色空間の複数フレームバッファを取り込み、接続ディスプレイの色空間に正規化してからスキャンアウト
- AGXおよびAVDが作成した圧縮フレームバッファの直接スキャンアウト
- 混合ダイナミックレンジコンテンツの自動正規化

これらすべてがDCPの「プレーン」の概念に結びついています。最初はリファクタリングなしでY’CbCrサポートを追加しようとしましたが、swap requestの構築方法と統合するのが非常に煩雑で複雑でした。プレーンコードのリファクタリングにより、Y’CbCrサポートの追加とswap request構築の両方がシンプルになりました。

これにより**非常に初期段階**のHDR実験を開始でき、Y’CbCrビデオソースを含むより完全なオーバーレイサポートも実現できました。Plasma 6.5にはオーバーレイプレーンの非常に基本的なサポートがfeature flagの裏に隠されていますが、まだかなり壊れています。関連するKWinのバグのいくつかはPlasma 6.7で修正予定で、これによりDCPのオーバーレイサポートをさらに拡張できるかもしれません。

さらにOliverさんは圧縮フレームバッファサポートにも着手しました。Apple Silicon SoCで使われている独自のAppleフレームバッファフォーマットは現在2つ知られています。AGX独自のフォーマットはすでにMesaでサポートされていますが、macOSは実際にDCPにこのフォーマットでフレームバッファを送りません。代わりにGPUレンダリングフレームバッファもAVDデコードビデオも常に「Apple Interchange」フォーマットでスキャンアウトされます。Oliverさんがこの新フォーマットをリバースエンジニアリングし、MesaとDCPドライバに実験的なサポートを追加しました。まだ開発中ですが、将来的にビデオ視聴などのディスプレイ負荷の高いタスクでメモリ帯域と消費電力を大幅に削減できるはずです。DCPとファームウェアの実験から、AGXフォーマットの直接読み込みも可能かもしれないことが示唆されていますが、macOSの観察に頼れないため、さらに調査が必要です。

またLinaさんがAGXのリバースエンジニアリング中に、macOSがInterchangeフレームバッファをシェーダーで解凍しているのを観察したため、一部のAGXバリアントはこのフォーマットに対応していない可能性があります。その場合、AVDデコードビデオストリームのみInterchangeを使い、DCPがAGXフォーマットをサポートしている場合はそちらにフォールバック、それ以外はGPUレンダリングコンテンツに対してはリニアフレームバッファにフォールバックすることになります。

新機能追加以外にも、プレーン処理コードのリワークにより、内蔵MacBookディスプレイの過飽和色問題をkernel 6.18以降でより簡単に修正できるようになりました。現在ICCプロファイルで回避している人は、これを無効にしてください。DCPの内部色処理と競合します。

プレーンはパズルの一片に過ぎません。ドライバのクリーンアップとHDRなどの機能をリリース可能な状態にする作業はまだまだたくさんあります。お楽しみに！

## Say cheese!（チーズ！）

ウェブカメラサポートを出荷してからかなり経ちましたが、**ほとんどの**ユーザーでは「普通に動いた！」という感じです。でも**全員**ではありません。

特にGNOMEのCameraアプリなど一部のウェブカメラアプリのユーザーが、初日から深刻な問題を報告していました。最初はGNOMEアプリ側の問題かと思われましたが、そうではありませんでした。Asahi OpenGLドライバがプラナー形式のビデオを不適切に扱っていたのです。ISP/ウェブカメラはV4L2経由でプラナー形式のビデオフレームバッファを出力するため、これをRGBフレームバッファに変換してデスクトップと合成する必要があります。GNOME CameraのようなアプリはGPUを使ってこれを行うため、激しく失敗していました。この[修正](https://gitlab.freedesktop.org/mesa/mesa/-/merge_requests/38160)を調べている過程でJanneさんがHoneykrispがプラナー形式のプレーン数を正しく通知していないことに気づき、[それも修正](https://gitlab.freedesktop.org/mesa/mesa/-/merge_requests/38200)しました。デバッグ中にRobert MaderさんがFedoraがGStreamerのgtk4paintablesinkプラグインをY’CbCrサポート付きでビルドしていなかったことを発見し、これはFedora Linux 43で修正されます。

これで全部解決？ ……いいえ！ GPUドライバのバグの裏に、さらにPipeWireの2つのバグが隠れていました。1つ目はPipeWireのGStreamerコードの整数オーバーフローで、Robertさんが[修正](https://gitlab.freedesktop.org/pipewire/pipewire/-/merge_requests/2582)しました。これで2つ目のバグが露呈：ストリームのレイテンシを決定するコードがperiod numeratorを常に1と仮定していましたが、Apple Siliconでは256/7680（30 fpsに相当）で表現されるため、正規化されず、極端に長いレイテンシになってPipeWireからのデータ待ちでストリームがクラッシュしていました。Janneさんが[修正MR](https://gitlab.freedesktop.org/pipewire/pipewire/-/merge_requests/2636)を提出し、PipeWire 1.4.10に取り込まれました。なぜ256/7680が1/30に簡約されないのかは謎のままですが、少なくともこの2つのパッチで大丈夫……ですよね？

……グラフィックスプログラミングは本当に難しいですね。実はGPUカーネルドライバが外部デバイスからのDMA-BUFを適切に扱っておらず、インポートしたバッファの使用後にデッドロックしていました。これを修正し、毎フレームトリガーされる非常にうるさいログメッセージを削除したところ、ウェブカメラが完全に生き返りました！ これでほとんどのアプリケーションでウェブカメラが完全にサポートされたはずです。

## The beast stirs（獣が動き出す）

過去12ヶ月で上流へのパッチ適用が驚異的な進展を見せました。パッチセットは6.13.8の1232個から6.18.8では858個に縮小。コード行数の差分も95,000行から83,000行に減少しました。パッチ数が30%減ったのにコード行数が15%しか減っていないのはちょっとおかしい……？

すべてのパッチが同等ではありません。上流に取り込まれたパッチには小さな修正もあれば、数千行のものもあります。しかしどれもGPUドライバに比べればかすんで見えます。

GPUドライバ単体で21,000行（まだ持っているダウンストリームRust抽象を除く）。DCPドライバのほぼ2倍、ISP/ウェブカメライバの3倍の大きさです。そして上流化作業が**ついに始まりました**。

かなり前にDRMメンテナから「ドライバ本体なしでUAPIヘッダだけ先に上流化してもいいよ（ただしその後ドライバが続くこと）」という特別許可をいただいていました。Janneさんが今、DRMドライバのテストスイートである[IGTへのパッチ](https://lore.kernel.org/asahi/20260105-asahi-tests-wave1-v1-0-a6c72617e680@jannau.net/T/#t)でその基盤を築いています。

ドライバを上流可能な状態にするためのクリーンアップ作業がまだ残っており、サイズが大きいためレビューにはかなり時間がかかるでしょう。近日中にこの方面で良いニュースをお届けできることを願っています！

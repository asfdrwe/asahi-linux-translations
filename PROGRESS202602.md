# 進捗報告：Linux 6.19

遅くなりましたが、あけましておめでとうございます！  
Linux 6.19がリリースされました……はい、早速本題に入ります。皆さんが知りたいことはわかっています。

## 大ニュース

Asahi Linux は今年で5周年を迎えます。この5年間で、シリアルポート経由の『Hello World』から、Linux エコシステムで最高レベルのサポートを受けた AArch64 デスクトップ
品質のプラットフォームの一つへと成長しました。Asahi Linux への継続的な注目が、多くの開発者に AArch64 を本気で取り組むきっかけを与えました。人気ソフトウェアの
プラットフォーム固有バグが次々と修正され、Apple Silicon 上の Linux で使えるようになりました。これまでの成果を非常に誇りに思っており、このプロジェクトは
大成功であり続けています。

それでも、誰もが口にする一つの質問がずっとありました。  
あらゆる発表でも上流への統合成功でもブログ投稿でも必ずと言っていいほど出てくる質問です。IRC や Matrix では週に最低1回は聞かれ、メールでも時々届きます。

『USB-C 経由のディスプレイ出力はいつ対応ですか？』  
『DisplayPort Alt Mode の予定は？』  
『MacBook Air で HDMI アダプタ使えるようになりましたか？』

特定の機能の対応予定時刻を聞かないでくださいと何度も丁寧にお願いしたにも関わらず、質問は止みませんでした。対策として『最低でもこの日付』と設定し、質問が来るたびに
倍にする案も出ましたが、すぐに宇宙の熱的死滅予測日を超えてしまいました。結局、id Software 方式の定番回答に戻りました。

DP Alt Mode　は完了したときに完了です。

そして……完了しました。ある程度ですが。

12月にSven 氏が[39C3で講演](https://media.ccc.de/v/39c3-asahi-linux-porting-linux-to-apple-silicon)を行い、これまでの Asahi の物語、リバース
エンジニアリングの過程、そして近未来について語りました。講演の最後に、スライドが M1 MacBook Air 上で動作しており、USB-C → HDMI アダプタ経由で会場の A Vシステムに
接続されていたことを明かしました！

同時に、[fairydustブランチ](https://github.com/AsahiLinux/linux/tree/fairydust)を下流 Linux ツリーで静かに公開しました。このブランチは Sven氏、Janne氏、marcan氏 に
よる長年の苦労の結晶で、このプラットフォームの脆く複雑なUSB・ディスプレイスタックを何とか制御・飼いならしたものです。Apple SiliconのUSB-Cポートからディスプレイ信号を出すには、
DCP[^1]、DPXBAR[^2]、ATCPHY[^3]、ACE[^4] の 4 つの異なるハードウェアブロックが必要です。それぞれにリバースエンジニアリング、Linux ドライバ作成、そしてお互いが
協調動作するように調整が必要でした。

とはいえ、まだやるべきことはあります。  
現在の fairydust ブランチでは、特定の USB-C ポートだけを DisplayPort 用に『有効化』しているため、複数の USB-C ディスプレイ はまだ使えません。
コールドプラグ・ホットプラグ時の挙動にもクセがあります。また、一部のユーザーが報告しているように、特定のディスプレイ設定で DCP が正しく動作せず、
色が不正確・過飽和になったり、タイミングモードが欠落したりする問題もあります。

こうした理由から、fairydust ブランチは現状のまま提供します。主に開発者がこれらの問題を最小限の対応で解決する手助けになることを目的としています。Apple Silicon　で
自分でカーネルをビルド・インストールできる方はぜひお試しください。ただし、一般向けに準備ができるまではサポートはできません。

## もう一つの大ニュース

かなり前から m1n1 には M3 シリーズへの基本対応が入っていました。足りなかったのは、各機種ごとの DeviceTree と、M3 特有のハードウェア挙動や M2 からの変更に
対応した Linux カーネルパッチでした。既存のパッチセットが扱いやすくなってから本格的に取り組む予定でしたが、基礎作業が新しい貢献者を刺激して手伝ってくれることを
密かに期待していました。結果、*3人*の新しい貢献者が現れました！

[Alyssa Milburn 氏 (noopwafel)](https://noopwafel.net)、[Michael Reeves 氏 (integralpilot)](https://github.com/IntegralPilot)、[Shiz 氏](https://shiz.me)の
3 人が Janne 氏の助けを借りて予備的な DeviceTree を作成し、驚くほど多くのハードウェアがほとんど変更なしで動くことを確認しました。NVMe と割り込みコントローラに軽微なカーネル修正を
加えたところ、Michael 氏が M3 MacBook Air で[KDE Plasmaまでブート](https://www.reddit.com/r/AsahiLinux/comments/1qnddjd/m3_now_has_fedora_43_asahi_remix_working_with_kde/)に成功しました！

つまり現在の M3 対応状況は、最初の Arch Linux ARM ベータを出した頃の M1 対応状況とほぼ同等です（キーボード、タッチパッド、WiFi、NVMe、USB3 が動作。ただし m1n1 と
Asahi カーネルにまだ統合されていないのでローカルパッチが必要）。ではもうすぐリリースですか？そう簡単ではありません。

5 年で状況は大きく変わりました。私たちは『最も完成度が高く洗練された AArch64 デスクトップLinux体験』として、また一般的なデスクトップ Linux としても
*トップクラス*の評価を得ています。この評価はとても誇らしいもので、多くの個人的な尽力の上に成り立っています。無駄にしたり当たり前だと思ったりするつもりはありません。

理想的には、現在の M1 / M2 対応レベルが M3 の一般公開の最低ラインであるべきです。でもそれは現実的でないですし、昔の初期 ALARM のような中途半端なものを出すのも避けたいです。
ではリリースまでに何が必要でしょうか？非常に多くあります。

まず最初に気づくのは、グラフィックが完全ソフトウェアレンダリングであることです。大変遅く、消費電力も大きく、ターミナルでテキストをスクロールするのもやっとです。
残念ながらこれは当面改善しそうにありません。M3 シリーズの GPU は M1 / M2 と大きく異なり、ハードウェアレイトレーシング、メッシュシェーダー、Dynamic Caching などが
追加されています。Alyssa 氏と Michael 氏が M3 GPU のリバースエンジニアリングに取り組み、dougallj 氏や TellowKrinkle 氏の成果を基に M2 から M3 への GPU ISA 変更に
関して多くの進展を見せています。

また、現在は iBoot が DCP を初期化しフレームバッファを提供してもらう形になっており、自分たちで DCP を正しく駆動できていません。これが非常に遅く非効率で、
バックライトなどの多くのディスプレイ機能がまともに制御できません。M3 は macOS 13.5 で動かず、macOS 14 で DCP ファームウェアインターフェースが変わっているため、
さらにリバースエンジニアリングが必要です。幸いプロトコル部分は変わっていないので、既存ツールを少し修正するだけで新しいインターフェースをトレースできます。

ハードウェア有効化以外にも、Asahi にふさわしい多くの仕上げ作業が必要です。Energy-Aware Scheduling、スピーカー保護 ＆ EQ 調整、マイク・ウェブカメラ対応など、
まだない機能がたくさんあるので、しばらくかかります。このうち Energy-Aware Scheduling はリリースを止めない品質管理機能ですが、speakersafetyd での M3 対応などは
リリースブロッカーになっています。

M3 対応が公開可能になるまで*それほど*長くはかからないと期待していますが、他のすべてと同じくETAはお答えできませんし、聞かないでくださいね。

以下は、前の翻訳の続き（M3サポートに関する期待の部分以降）からの完全な日本語翻訳です。元の記事のトーンを保ち、自然な日本語で技術用語を適切に訳しています。

## 周波数良好

14 インチおよび 16 インチの MacBook Pro は非常に優れたディスプレイを搭載しています。色再現性が極めて正確で、非常に明るく、*しかも 120 Hzのリフレッシュレートに対応*しています。
ただし、ひとつ落とし穴があります。

macOS では、単にディスプレイを 120 Hzに設定して終わり、というわけにはいきません。Apple は 60 Hzを超えるリフレッシュレートを『ProMotion』という機能の裏に隠しています。
これは実質的に普通の可変リフレッシュレート（Variable Refresh rate, VRR）のマーケティング用語に過ぎません。macOS 特有のクセだと思い込み、DCP ファームウェアで 120 Hzの
タイミングモードを選べば Linux でもそのレートで駆動できるだろうと考えるのは無理もないですが、実際にはそうではありません。

Apple のみが分かる理由により、DCP は surface swap request 構造体の特定の3つのフィールドが埋められていない限り、MacBook Pro のパネルを 60 Hz超で駆動することを拒否します。
これらのフィールドが何らかのタイムスタンプであることは以前からわかっていましたが、それ以上深く調べる時間がありませんでした。そこで、またしても新しい貢献者が登場！

[Oliver Bestmann 氏](https://github.com/oliverbestmann)が自ら MacBook Pro で 120 Hzを実現することを目標に、これらの3つのタイムスタンプを調査しました。
macOS のトレースを分析したところ、これらは CPU タイマーのティック単位で増加しており、ほぼ常に 1 フレーム間隔であることがわかりました。これはフレーム提示時刻管理に
使われていることを示唆しています。提示時刻管理は VRR を正しく動作させるために必要で、コンポジタとドライバの両方が、特定のフレームが実際にディスプレイに表示された
時刻を知る必要があります。VRRが無効でも、コンポジタはこの情報を使ってフレームペーシングを安定させ、ティアリングを最小限に抑えることができます。

現時点では一貫した 120 Hzだけを目標としており、VRR はまだ対象外です。macOS が両者を密接に結びつけているため、DCP が 120 Hzのために何を想定しているのかを正確に把握するのは困難です。
タイムスタンプが必要なのは明らかですが、理由は？ DCP は何をするのでしょうか？ そして*正確には*何を表しているのでしょうか？

時にはバカなことをするのが賢い選択になります。Oliver 氏はタイムスタンプは VRR でのみ*意味がある*と仮定し、各タイムスタンプフィールドに静的な値を詰め込んでみました。
すると……動きました！ kernel 6.18.4 以降、14 インチ/ 16 インチMacBook Pro の所有者は内蔵ディスプレイを 120 Hzで駆動できるようになりました。

もちろん、この解決策は明らかにジャンク(jank)です。提示タイムスタンプは現在、KMS サブシステムが atomic state flush をトリガーするたびに設定されており、
静的な値にするのは本来想定されていません。この方法は私たちのユースケースでは機能しますが、VRR 対応を不可能にします。これが次の話題につながります。

## 着陸開始

Linux 向け DCP ドライバはこれまでかなり不完全でした。これは驚くことではありません。ディスプレイエンジンは極めて複雑で、これが反映される DCP が動作するファームウェアは
9 MiB もの巨大なブロブです。このファームウェアは macOS と密接に統合されるよう設計されたインターフェースを公開しており、macOS リリース間で破壊的に変更されるため、
バージョン付き構造体や関数呼び出しの特別な扱いが必要です。

これらが積み重なって、ドライバは最適とは言えない部分的な形で開発されてきました。理由はたくさんあります：

- 時間的余裕が不足（特に Janne 氏は Asahi カーネルツリーの管理と再構成の負担を一手に引き受け）
- 他のハードウェアの立ち上げなどもっと優先度の高い作業が存在
- 元々ファームウェアのバージョン管理をより良く扱うために Rust でドライバを書き直す予定

これらに加えて、当時の設計目標では単に問題ではなかったのです。初期の目標はラップトップの内蔵ディスプレイとデスクトップの HDMI ポートを確実に駆動できる程度の DCP を立ち上げることでした。
それを達成するために、DCPのファームウェアインターフェースのごく一部を KMS API に無理やりくっつけて、各スワップで単一の 8 ビット ARGBフ レームバッファをスキャンアウトできるように
しただけです。

その後、DisplayPort / HDMI 経由の音声対応、Night Light 向けの基本的な色管理（Color Transformation Matrices 対応）、初歩的なハードウェアオーバーレイを実装しました。
しかし、HDR、VRR、他のフレームバッファフォーマット対応、外部ディスプレイのハードウェア輝度制御（DDC/CI）、マルチメディアやフルスクリーンアプリ向けの直接スキャンアウトなど、
まだ多くの機能が残っています。

現在のドライバアーキテクチャの枠内でこれらに対応するのは困難です。ユーザースペースとの統合や KMS API との一部コンポーネントのインタラクトに未解決の問題がいくつもあります。
それでも新機能を進めたいと考えており、Rust KMS バインディングが上流に統合されるのを待っていると非常に時間がかかるため、必要に応じて既存 DCP ドライバの部分的なリファクタリングを
開始しました。最初に取りかかったのはハードウェアプレーン(hardware plane)処理のコードです。

なぜプレーンから？ ハードウェアプレーンの適切な対応は性能と効率にとって重要です。ほとんどのディスプレイエンジンは複数のフレームバッファをハードウェアで合成する機能を持っており、
DCP も例外ではありません。レイヤリング、移動、ブレンド、基本的な色変換が可能です。典型的なユースケースはカーソルです。カーソルが動くたびに GPU がデスクトップ全体を再描画する
代わりに、カーソルをディスプレイエンジンのオーバーレイプレーンの1つに置き、静的なフレームバッファを画面上移動させるだけにします。GPUは画面上の**コンテンツ**が再描画を
必要とする場合（ボタンにホバーしたときなど）にのみアクティブにレンダリングします。

以前に極めて限定的な対応を無理やり入れたところ、Plasma 6 のハードウェアカーソル対応と上手く動作していました。しかしもっと深く掘り下げる必要があります。

DCP は非常に便利な機能を持っており、その一部は HDR や直接ビデオスキャンアウトに絶対必要です。特に重要なのは：

- セミプレーン Y’CbCr フレームバッファ（SDR および HDR）の直接スキャンアウト
- 異なる色空間の複数フレームバッファを取り込み、接続ディスプレイの色空間に正規化してからのスキャンアウト
- AGX[^5] および AVD[^6] が作成した圧縮フレームバッファの直接スキャンアウト
- 混合ダイナミックレンジコンテンツの自動正規化

これらすべてが DCP のプレーンの概念に結びついています。最初はリファクタリングなしで Y’CbCr 対応を追加しようとしましたが、swap request の構築方法と統合するのが
非常に煩雑で複雑でした。プレーンコードのリファクタリングにより、Y’CbCr 対応の追加と swap request 構築の両方がシンプルになりました。

これにより*非常に初期段階*の HDR 実験を開始でき、Y’CbCr ビデオソースを含むより完全なオーバーレイ対応も実現できました。Plasma 6.5 にはオーバーレイプレーンの
非常に基本的な対応が feature flag の裏に隠されていますが、まだかなり壊れています。関連する KWin のバグのいくつかは Plasma 6.7 で修正予定で、これにより DCP の
オーバーレイ対応をさらに拡張できるかもしれません。

さらに Oliver 氏は圧縮フレームバッファ対応にも着手しました。Apple Silicon SoC で使われている独自の Apple フレームバッファフォーマットは現在 2 つ知られています。
AGX 独自のフォーマットはすでに Mesa で対応していますが、macOS は実際には DCP にこのフォーマットでフレームバッファを送りません。代わりに GPU レンダリングフレームバッファも
AVD デコードビデオも常に『Apple Interchange』フォーマットでスキャンアウトされます。Oliver 氏がこの新フォーマットをリバースエンジニアリングし、Mesa と DCP ドライバに
実験的な対応を追加しました。まだ開発中ですが、将来的にビデオ視聴などのディスプレイ負荷の高いタスクでメモリ帯域と消費電力を大幅に削減できるはずです。DCP とファームウェアの実験から、
AGX フォーマットの直接読み込みも可能かもしれないことが示唆されていますが、macOS の観測に頼れないため、さらに調査が必要です。

また Lina 氏が AGX のリバースエンジニアリング中に、macOS が Interchange フレームバッファをシェーダーで伸長しているのを観測したため、一部の AGX バリアントはこのフォーマットに
対応していない可能性があります。その場合、AVD デコードビデオストリームのみ Interchange を使い、DCP が AGX フォーマットに対応している場合はそちらにフォールバック、それ以外は
GPU レンダリングコンテンツに対してはリニアフレームバッファにフォールバックすることになります。

新機能追加以外にも、プレーン処理コードのリワークにより、内蔵 MacBook ディスプレイの過飽和色問題を kernel 6.18 以降でより簡単に修正できるようになりました。現在 ICC プロファイルで
回避している方はこれを無効にしてください。DCP の内部色処理と競合します。

プレーンはパズルの一片に過ぎません。ドライバのクリーンアップと HDR などの機能をリリース可能な状態にする作業はまだまだたくさんあります。お楽しみに！

## チーズ！

ウェブカメラ対応を公開してからかなり経ちましたが、*ほとんどの*ユーザーでは普通に動いた！という感じです。でも*全員*ではないですよ。

特に GNOME の Camera アプリなど一部のウェブカメラアプリのユーザーが初日から深刻な問題を報告していました。最初は GNOME アプリ側の問題かと思われましたが、そうではありませんでした。
Asahi OpenGL ドライバがプラナー形式のビデオを不適切に扱っていたのです。ISP / ウェブカメラは V4L2 経由でプラナー形式のビデオフレームバッファを出力するため、これを RGB フレーム
バッファに変換してデスクトップと合成する必要があります。GNOME Camera のようなアプリは GPU を使ってこれを行うため、激しく失敗していました。
この[修正](https://gitlab.freedesktop.org/mesa/mesa/-/merge_requests/38160) を調べている過程で Janne 氏が Honeykrisp がプラナー形式のプレーン数を正しく通知
していないことに気づき、[それも修正](https://gitlab.freedesktop.org/mesa/mesa/-/merge_requests/38200)しました。デバッグ中に Robert Mader 氏が Fedora が
GStreamer の gtk4paintablesink プラグインを Y’CbCr 対応付きでビルドしていなかったことを発見しました。これは Fedora Linux 43 で修正されます。

これで全部解決？ ……いいえ！ GPU ドライバのバグの裏に、さらに PipeWire の 2 つのバグが隠れていました。1 つ目は PipeWire の GStreamer コードの整数オーバーフローで、
Robertさんが[修正](https://gitlab.freedesktop.org/pipewire/pipewire/-/merge_requests/2582)しました。これで 2 つ目のバグが露呈：ストリームのレイテンシを
決定するコードが period numerator を常に 1 と仮定していましたが、Apple Silicon では 256 / 7680（30 fpsに相当）で表現されるため、正規化されず、極端に長い
レイテンシになって PipeWire からのデータ待ちでストリームがクラッシュしていました。Janne 氏が[修正マージリクエスト](https://gitlab.freedesktop.org/pipewire/pipewire/-/merge_requests/2636)を提出し、PipeWire 1.4.10 に取り込まれました。なぜ 256 / 7680 が 1/30 に簡約されないのかは謎のままですが、少なくともこの 2 つのパッチで大丈夫……ですよね？

グラフィックスプログラミングは本当に難しいです。実は GPU カーネルドライバが外部デバイスからの DMA-BUF を適切に扱っておらず、インポートしたバッファの使用後にデッドロックしていました。
これを修正し、毎フレームトリガーされる非常にうるさいログメッセージを削除したところ、ウェブカメラが完全に生き返りました！ これでほとんどのアプリケーションでウェブカメラへ完全に
対応したはずです。

## 獣が動き出す

過去12ヶ月で上流へのパッチ適用が驚異的な進展を見せました。パッチセットは 6.13.8 の 1232 個から 6.18.8 では 858 個に縮小。コード行数の差分も 95,000 行から 83,000 行に
減少しました。パッチ数が 30 %減ったのにコード行数が 15 %しか減っていないのはちょっと変ですね……

すべてのパッチが同等ではありません。上流に取り込まれたパッチには小さな修正もあれば、数千行のものもあります。しかしどれも GPU ドライバに比べればかすんで見えます。

GPU ドライバ単体で 21,000 行（まだ持っている下流 Rust 抽象を除く）です。DCP ドライバのほぼ2倍、ISP / ウェブカメライバの 3 倍の大きさです。そして上流化作業がついに始まりました。

かなり前に DRM メンテナから「ドライバ本体*なしで* UAPI ヘッダだけ先に上流化してもいいよ（ただしその後ドライバが続くこと）」という特別な許可をいただいていました。
Janne 氏が今 DRM ドライバのテストスイートである[IGTへのパッチ](https://lore.kernel.org/asahi/20260105-asahi-tests-wave1-v1-0-a6c72617e680@jannau.net/T/#t)で
基盤を講師クしています。

ドライバを上流可能な状態にするためのクリーンアップ作業がまだ残っています。サイズが大きいためレビューにはかなり時間がかかるでしょう。近日中にこの方面で良いニュースをお届けできることを願っています！

# 進捗報告：Linux 6.19

遅くなりましたが、あけましておめでとうございます！  
Linux 6.19がリリースされました……はい、早速本題に入ります。皆さんが知りたいことはわかっています。

## 大ニュース

Asahi Linux は今年で5周年を迎えます。この5年間で、シリアルポート経由の『Hello World』から、Linux エコシステムで最高レベルのサポートを受けた AArch64 デスクトップ
品質のプラットフォームの一つへと成長しました。Asahi Linux への継続的な注目が、多くの開発者に AArch64 を本気で取り組むきっかけを与えました。人気ソフトウェアの
プラットフォーム固有バグが次々と修正され、Apple Silicon 上の Linux で使えるようになりました。これまでの成果を非常に誇りに思っており、このプロジェクトは
大成功であり続けています。

それでも、誰もが口にする一つの質問がずっとありました。  
あらゆる発表でも上流への統合成功でもブログ投稿でも必ずと言っていいほど出てくる質問です。IRC や Matrix では週に最低1回は聞かれ、メールでも時々届きます。

『USB-C 経由のディスプレイ出力はいつ対応ですか？』  
『DisplayPort Alt Mode の予定は？』  
『MacBook Air で HDMI アダプタ使えるようになりましたか？』

特定の機能の対応予定時刻を聞かないでくださいと何度も丁寧にお願いしたにも関わらず、質問は止みませんでした。対策として『最低でもこの日付』と設定し、質問が来るたびに
倍にする案も出ましたが、すぐに宇宙の熱的死滅予測日を超えてしまいました。結局、id Software 方式の定番回答に戻りました。

DP Alt Mode　は完了したときに完了です。

そして……完了しました。ある程度ですが。

12月にSven 氏が[39C3で講演](https://media.ccc.de/v/39c3-asahi-linux-porting-linux-to-apple-silicon)を行い、これまでの Asahi の物語、リバース
エンジニアリングの過程、そして近未来について語りました。講演の最後に、スライドが M1 MacBook Air 上で動作しており、USB-C → HDMI アダプタ経由で会場の A Vシステムに
接続されていたことを明かしました！

同時に、[fairydustブランチ](https://github.com/AsahiLinux/linux/tree/fairydust)を下流 Linux ツリーで静かに公開しました。このブランチは Sven氏、Janne氏、marcan氏 に
よる長年の苦労の結晶で、このプラットフォームの脆く複雑なUSB・ディスプレイスタックを何とか制御・飼いならしたものです。Apple SiliconのUSB-Cポートからディスプレイ信号を出すには、
DCP[^1]　、**DPXBAR**、**ATCPHY**、**ACE** の4つの異なるハードウェアブロックが必要です。それぞれにリバースエンジニアリング、Linuxドライバ作成、そしてお互いが仲良く動くよう調整が必要でした。

とはいえ、まだやるべきことはあります。  
現在のfairydustブランチでは、特定のUSB-CポートだけをDisplayPort用に「有効化」しているため、複数のUSB-Cディスプレイはまだ使えません。コールドプラグ・ホットプラグ時の挙動にもクセがあります。また、一部のユーザーが報告しているように、特定のディスプレイ設定でDCPが正しく動作せず、色が不正確・過飽和になったり、タイミングモードが欠落したりする問題もあります。

こうした理由から、fairydustブランチは**現状のまま**提供します。主に開発者がこれらの問題を最小限のサポートで解決する手助けになることを目的としています。Apple Siliconで自分でカーネルをビルド・インストールできる方はぜひ試してみてください。ただし、一般向けに準備ができるまではサポートはできません。

## もう一つの大きなニュース

かなり前からm1n1にはM3シリーズの基本サポートが入っていました。足りなかったのは、各機種ごとのDevicetreeと、M3特有のハードウェア挙動やM2からの変更に対応したLinuxカーネルパッチでした。既存のパッチセットが扱いやすくなってから本格的に取り組む予定でしたが、基礎作業が新しい貢献者を刺激して手伝ってくれることを密かに期待していました。結果、**3人**の新しい貢献者が現れました！

[Alyssa Milburn (noopwafel)](https://noopwafel.net)、[Michael Reeves (integralpilot)](https://github.com/IntegralPilot)、[Shiz](https://shiz.me)の3人がJanneの助けを借りて予備的なDevicetreeを作成し、驚くほど多くのハードウェアがほとんど変更なしで動くことを確認しました。NVMeと割り込みコントローラに軽微なカーネル修正を加えたところ、MichaelがM3 MacBook Airで[KDE Plasmaまでブート](https://www.reddit.com/r/AsahiLinux/comments/1qnddjd/m3_now_has_fedora_43_asahi_remix_working_with_kde/)に成功しました！

つまり現在のM3サポート状況は、最初のArch Linux ARMベータを出した頃のM1サポートとほぼ同等です（キーボード、タッチパッド、WiFi、NVMe、USB3が動作。ただしm1n1とAsahiカーネルにまだマージされていないローカルパッチが必要）。ではもうすぐリリースですか？ ……というと、そう簡単ではありません。

5年で状況は大きく変わりました。私たちは「最も完成度が高く洗練されたAArch64デスクトップLinux体験」として、また一般的なデスクトップLinuxとしてもトップクラスの評価を得ています。この評価はとても誇らしいもので、多くの個人的な犠牲の上に成り立っています。無駄にしたり当たり前だと思ったりするつもりはありません。

理想的には、現在のM1/M2サポートレベルがM3の一般リリースの最低ラインであるべきです。でもそれは現実的ではなく、昔の初期ALARMのような中途半端なものを出すのも避けたいです。ではリリースまでに何が必要か？　かなりたくさんあります。

まず最初に気づくのは、グラフィックが**完全ソフトウェアレンダリング**であることです。非常に遅く、消費電力も大きく、ターミナルでテキストをスクロールするのもやっとです。残念ながらこれは当面改善しそうにありません。M3シリーズのGPUはM1/M2と大きく異なり、ハードウェアレイトレーシング、メッシュシェーダー、Dynamic Cachingなどが追加されています。AlyssaとMichaelがM3 GPUのリバースエンジニアリングに取り組み、dougalljやTellowKrinkleの成果を基にM2→M3のGPU ISA変更にすでに進展を見せています。

また、現在はiBootにDCPを初期化させてフレームバッファをもらう形になっており、自分たちでDCPを正しく駆動できていません。これが非常に遅く非効率で、バックライトなどの多くのディスプレイ機能がまともに制御できません。M3はmacOS 13.5が動かず、macOS 14でDCPファームウェアインターフェースが変わっているため、さらにリバースエンジニアリングが必要です。幸いプロトコル部分は変わっていないので、既存ツールを少し修正するだけで新しいインターフェースをトレースできます。

ハードウェア有効化以外にも、Asahiらしい多くの仕上げ作業が必要です。Energy-Aware Scheduling、スピーカー保護＆EQ調整、マイク・ウェブカメラ対応など、まだない機能がたくさんあり、しばらくかかります。一部（例：Energy-Aware Scheduling）はリリースを止めないQoL機能ですが、speakersafetydでのM3対応などはリリースブロッカーです。

M3サポートが出荷可能になるまで**それほど**長くはかからないと期待していますが、他のすべてと同じくETAはお答えできませんし、聞かないでくださいね。

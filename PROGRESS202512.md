# 進捗報告: Linux 6.18

Linux 6.18 が公開されました。ホリデーシーズンの進捗報告ではお話しすることがちょっとだけありますよ！

## 上流への統合が更に進む

前回、コア SMC ドライバーが3年の長い歳月を経てようやく上流に統合されたことを発表しました。その成功に続き、SMC のサブデバイスドライバーを統合するプロセスを開始しました。
これらのドライバーは、SMC のすべての機能をさまざまなカーネルサブシステムに統合します。hwmon ドライバーはすでに 6.19 向けに受け入れられており、SMC が制御する
多数の電圧、電流、温度、電力センサーを標準の hwmon インターフェースで読み取れるようになります。SMC は RT Cの読み取りと設定も担当しており、この機能のドライバーも 6.19 向けに
統合されました！ 統合待ちの SMC サブデバイスは、電源ボタンと蓋スイッチのドライバ（まだメーリングリスト上）と、バッテリー/電源管理ドライバで、現在、macOS 26 の SMC 
ファームウェア変更に対応するための調整が必要です。

また、USB-C ポート経由での USB3 対応に必要な変更もようやく上流に到達しました。これも長いプロセスで、当初下流で開発したものから大幅にアプローチを変更する必要がありました。
Synopsys USB コントローラーと TI USB-PD コントローラードライバーの変更は統合され、Apple Type-C PHY (ATCPHY)のドライバだけがレビュー待ちです。ATCPHY は
物理的な USB-C 接続の設定、必要なプロトコルの検出とネゴシエーション、関連コントローラーへの信号ルーティングを担当します。これは USB3 だけでなく、USB-C ポートが扱う
他のすべてのプロトコルにも必要です…

## いつもは一行で済むのに…
ほとんどの人はフルマイク対応を楽しんでいますが、M2 Pro/Max MacBook をお持ちの方は取り残されていました。M2 Pro/Max は M1 Pro/Max とほぼ同一なので、マイクアレイを
制御する『Always-On』[^1] Processorにもこれは当てはまるはずだと考えられていました。残念ながら、一時期この仮定が成り立たないように見えました。何が問題なのでしょう？
RTKit のバグ？ AOP の初期化方法の変更？ 確かめる唯一の方法は、機器を購入することでした…
~~新品~~リファービッシュ 16 インチ MacBook Pro と新鮮な Gentoo インストールを手に、調査を始め、すぐに最初の大きな問題にぶつかりました。すべてが『なんとなく』動作するのです。
AOP　は正しくブートし、マイクエンドポイントは初期化され、ALSA　ドライバは正しいメタデータで　PCM　を登録します。しかし、その　PCM　を開こうとすると無言で失敗します。おっと。

こんなものをどうデバッグしましょうか？ m1n1 のゲストとして Linux を実行し、macOS のように 16 進ダンプを何時間も眺める？ それともカーネルのビルトインデバッグ機能、ftrace や
kgdb を使う？ 叡智とは利用可能なすべてのデバッグ施設を活用して失敗の正確なポイントを迅速かつ効率的に特定することです。智慧は `printk()` です。

そして、次の 8 時間はそうでした。`printk()`、コンパイル、ブート。`printk()`、コンパイル、ブート。おっと、間違ったブランチを追っていました。`printk()`、コンパイル、ブート。

結局、DMA コアの奥深くまで潜り込み、IOMMU コードの近くから脱出しました。メモリ割り当てが失敗しており、それが　ALSA　ドライバーまでバブルアップしていました。さらに少し追跡すると、
割り当て自体ではなく、メモリ領域を　I/O　仮想アドレスにマッピングする部分が失敗していることがわかりました。

さまざまな理由で、IOMMU コアは特定の IOMMU に対する有効アドレスの範囲に制限を課すことに対応しています。DART の場合、これらの範囲を Devicetree の `apple,dma-range` プロパティで
指定します。指定された範囲外の IOVA をマッピングしようとすると、ドライバーはエラーを返します。このプロパティの値は Apple の Devicetre eから来ますが、Linux の Devicetree ソースに
手動で入力する必要があります。もうおわかりですね…

`apple,dma-range` プロパティを削除して、もう一度ブートすると、ついに望んでいたものが得られました。M2 Pro/Max ラップトップで動作するマイク！ パッチをすぐにカーネルツリーに適用し、
バージョンをタグ付けしました。ディストリビューションがこのタグに基づくカーネルをリリースしたら、システムをアップデートしてください！

ただし、まだ作業が残っています。*正しい値*の `apple,dma-range` プロパティでも IOVA 割り当てが失敗します。ありがたいことに、このプロパティは実際には必須ではなく、他の SoC の 
Devicetree では AOP DART に対して指定されていないので、M2 Pro/Max MacBook ユーザーは少なくともこれを解決するまでマイクを楽しめます。

GitHub Sponsors と OpenCollective の寛大なサポーターのおかげで、灯りを維持するだけでなく、プラットフォームのリバースエンジニアリングとサポートに必要なハードウェアを購入できます。
ただし、これらの資金は無限のお金 glitch ではありません。

OpenCollective は主に払い戻しモデルで機能し、*すでに*購入したものの請求書を提出・承認されてから資金がリリースされます。つまり、すべての支出はポケットマネーで、
購入から払い戻しまで最大5週間かかります。MacBook は安くなく、生活の他の支出があります。つまり、突然機器を魔法のように用意して、遭遇したすべてのバグにすべてを投じて即座に
対応することはできません。優先順位をつけなければならず、非 Asahi のコミットメントも含めます。この待ち時間が長いため、みなさまの忍耐力を試していることは理解しております。

## より良いインストール体験
Asahi のディストリビューションが存在しなくなることを望んでいると半分冗談で常に言ってきました。根本的には Asahi Linux は常にリバースエンジニアリングプロジェクトです。『公式』Asahi 
ディストリビューション、Asahi ALARM や Fedora Asahi Remix は、さまざまな上流プロジェクトと協力してコードを統合する間の暫定措置と常に考えられてきました。今では
Linuxカーネル、PipeWire、WirePlumber、Mesa での私たちの作業をみなさまが知っていますが、より目立たないが同等に重要なプロジェクトとも協力しています。

Apple Silicon Mac は本質的にスケールアップされた iPad で、SSD もそれを反映しています。これらのマシンは SSD にブートローダー、ファームウェア、工場出荷時キャリブレーションデータを保存し、
SEP がセキュリティに使用するデータを保存します。各マシンにはディスク末尾にシステムリカバリパーティションがあり、インストールされた各 macOS インスタンスに対応するリカバリ環境があります。
これらの*いずれか*を削除するとデータは終了し、DFU リカバリが必要になります。

これが Fedora Asahi Remix がディスクイメージ経由でインストールされる理由の一部です。インストーラが Mac のディスクを適切にパーティションし、関連パーティションに既知の良好なイメージを
フラッシュする方がはるかに安全です。このアプローチは単一ディストリビューションのパイプラインには最適ですが、深刻な欠点もあります。

まず、他のディストリビューションが単一プラットフォーム向けに特別なディスクイメージとインストーラーを構築するリリースエンジニアリングの負担を負うのは非現実的で不公平です。
これは比較的小さなユーザー基盤に対して多大な労力であり、各ディストリビューションそれぞれで重複する作業になってしまいます。

第二に、多くのユーザーにとって最も重要なのは、このアプローチは情報セキュリティの観点からあまりクリーンではありません。事前ロールされたディスクイメージをフラッシュすると、
インストール時にディスク暗号化を設定するのが非現実的になり、初回ブートでパーティション UUID を手動でスクランブルする必要があります。

理想の世界では、Apple Silicon Mac に Linux をインストールするのは、Asahi Installer でマシンを準備し、お気に入りのディストリビューションのライブイメージをブートして
そこからインストールするほど簡単であるべきです。Gentoo などの一部のディストリビューションはすでにこのアプローチを取っています。Gentoo はパーティションが完全に手動で、
ユーザーがほぼ常に文書をステップバイステップで従うため、これで対応可能です。ユーザー側の少しのガイダンスと注意でほとんどのパーティション事故を避けられます。しかし、
グラフィカルインストーラーを持つディストリビューションは通常、ディストリビューションのデフォルトでユーザーのディスクを自動パーティションすることを提案します。これらのツールは
主に PC 向けに設計されており、ターゲットディスク全体を喜んで一新します。当然これを許すわけにはいかず、発生する可能性が高いため、伝統的な主流のインストールアプローチにはまだ対応できません。

GUID Partition Table [^2]のすべてのパーティションはパーティションタイプ UUI Dでマークされます。これらは標準化されており、変わりません。Apple の特殊パーティションは
標準に含まれ、パーティション操作の回避ゾーンを簡単に識別できます。この目的で、Neal 氏が Anaconda チームと協力してこれらのパーティションを非表示にしました。これにより Anaconda の
ディスクパーティショニングモジュールで誤って削除されないようにします。この作業が完了しリリースに反映されれば、Anacondaを使用するディストリビューションはライブメディアから
安全にインストール可能になります！目標は Anaconda が Asahi Installer で作成された空き領域のみを表示し、恐れずに自動パーティションを可能にすることです。これにより、
ユーザーはフルディスク暗号化や非標準ファイルシステムなどのカスタムディスクセットアップを簡単に指定できます。

同様に、Neal 氏は KDE と協力して Plasma Setup という新しいオンボーディングアプリケーションを開発中です。これは初回ブート時のシステムセットアップを心地よい体験にすることを
目的としています。安定したら、現在の Calamares ベースの初回ブートウィザードを現代的で統合された体験に置き換えられます！

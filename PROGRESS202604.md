# 進捗報告：Linux 7.0

- [前回](https://github.com/asfdrwe/asahi-linux-translations/blob/main/PROGRESS202602.md)

約 3 年にわたる 6.x シリーズカーネルの開発を経て、ついに Linux 7.0 がリリースされました。これに伴い、Asahi Linux の新たな進捗報告をお届けします。

## すべてを自動化

代替ディストリビューションをご利用のユーザーや細かい部分に気づかれる方々は、Asahi Installer にいくつかの変更があったことに気がつかれたかもしれません。約 2 年ぶりに、Asahi Installer の更新版を CDN に公開しました。2 年という期間は更新として長すぎますが、これほど時間を要したのは何だったのでしょうか。

上流のインストーラーパッケージは少し複雑な仕組みになっています。インストーラーの大部分は　Python で記述されており、ブートストラップ用の小さな Bash スクリプトが組み合わされています。`curl | sh` を実行すると、まずブートストラップスクリプトがダウンロードされ、そこから実際のインストーラーバンドルが CDN から取得されます。このバンドルには、Python インタープリタと大幅に削減された標準ライブラリ、m1n1 ステージ 1 バイナリ、そしてインストーラー本体が含まれています。

これまでインストーラーのリリースを行うには、以下の手順が必要でした：

1. インストーラーリポジトリにタグ付け
2. macOS 用の Python ビルドをダウンロード
3. 特定のコミットから m1n1 をビルド
4. Python、m1n1、インストーラーをバンドル
5. インストーラーバンドルを CDN にアップロード
6. CDN のバージョン管理ファイルを更新

この作業は時間がかかり、CDN への管理者権限も必要でした。そのため、インストーラーの更新を長期間放置してしまい、前回のタグは 2024 年 6 月のままになっていました。上流への対応が進み、DeviceTree バインディングが頻繁に変更される中で、これはディストリビューションを保守する方々にとって大きな問題となっていました。

Asahi Installer には UEFI のみのインストールオプションがあります。このオプションは macOS を縮小し、UEFI 実行ファイルを起動するために必要な最小限のコンポーネント（m1n1 ステージ 1、DeviceTree、U-Boot）のみをインストールします。これにより、Asahi 対応のライブメディア（例：専用のGentoo Asahi LiveCD イメージ）から起動することが可能になります。

UEFI のみのインストールでは DeviceTree がインストーラーバンドル自体に含まれているので、カーネルが正常に起動するにはバンドルされた DeviceTree とカーネルとで想定する内容が一致している必要があります。しかし、上流化の過程で　DeviceTree バインディングが変更されたため、両者の同期が徐々に崩れていました。この問題はカーネル 6.18 で顕在化しました。Apple USB サブシステムに関する多数の変更が m1n1 と DeviceTree バインディングの両方に必要となり、ライブメディアからの 6.18 以降のカーネル起動が不可能になってしまったのです。

手作業で再度更新を行う代わりに、この機会に自動化を構築して問題を恒久的に解決することにしました。

インストール可能なイメージのマニフェストを[asahi-installer-dataリポジトリ](https://github.com/AsahiLinux/asahi-installer-data)に移動し、インストーラーのコードベースとは独立して更新できるようにしました。また、GitHub Actions を用いてインストーラーのデプロイも自動化しています。今後は、[asahi-installer](https://github.com/AsahiLinux/asahi-installer)の `main` ブランチへのプッシュごとに、インストーラーが自動的にビルドされ、`https://alx.sh/dev` にアップロードされます。GitHubにタグがプッシュされた場合は、`https://alx.sh` 向けに同じ処理が行われます。

最新版の 0.8.0 では、バンドルされる m1n1 ステージ 1 バイナリがバージョン 1.5.2 に更新され、Mac Pro への対応が追加されました。また、ファームウェア更新モードも導入されており、これは次に説明する機能と上手に連携します。

## どのように光センサーを過剰設計？

画面を搭載したほぼすべてのデバイスに何らかの光センサーが搭載されるようになりました。これは主に、周囲の明るさに応じて自動的に画面の輝度を調整するためのものです。スマートフォンのように外に出た際に画面が暗すぎて見えにくくなる状況では非常に便利な機能です。最もシンプルなものはフォトレジスタを使用しています。これは輝度変更だけを目的とする場合には十分ですが、周囲の照明条件が影響するのは輝度だけではありません。色再現はどうでしょうか？

Apple の機器では何年にも渡って『True Tone』ディスプレイ機能が搭載されています。この機能では、周囲の照明の明るさだけでなく、色特性*も*測定します。そして、そのデータに基づいて輝度と色の変換をディスプレイに適用し、常に可能な限り正確なコンテンツを表示できるようにします。これは特に演色評価数（Color Rendering Index）が低い照明（蛍光灯や安価な白色LEDなど）で顕著に効果を発揮します。この機能を実現する環境光センサーは、通常は I2C などの標準バスでシステムに接続される小さな IC です。これで基本的な用途には十分ですが、Apple の場合追加の考慮事項が色々あります：

- 光センサーは画面がオンになっている間常に動作するため、出力処理は可能な限り効率的であるべき
- 最高の精度を得るために校正が可能であるべき
- 複数の光センサーモデルが存在するが、OSはそれらを過度に意識する必要なし
- このプラットフォームの他のハードウェアと同様、光センサーにも3文字の略称（ALS）が必要

当然これは Always-On Processor（AOP）[^1]で実行するのにぴったりです。

chaos_princess 氏のおかげで、AOP と ALS のドライバはすでに動作していましたが、ALS から AOP が報告する生データは校正が存在しないとかなり不正確でした。この校正データはバイナリブロブであり、実行時に AOP にアップロードする必要があります。これは事実上のファームウェアです。Apple のバイナリを再配布できないため、インストール時に macOS から取得し、ドライバが参照できる場所に保存する必要があります。

これを実現するため、Asahi Installer は Linux で必要となるすべてのファームウェアを集め、作成する EFI システムパーティションに保存します。Dracut モジュールがこれを`/lib/firmware/`のサブディレクトリにマウントし、ドライバが参照できるようにしています。ただし、Asahi Linux インストール後に macOS から追加のファームウェアを取得する必要が生じた場合に問題が発生します。以前のウェブカメラの場合のように、ユーザーが手作業で EFI システムパーティションを修正する必要が出ないよう、chaos_princess 氏は Asahi Installer にファームウェアパッケージを自動更新する機能を追加しました。ALS を皮切りに、今後必要なファームウェア更新は、macOS または macOS Recovery を起動してAsahi Installer を再実行し、指示に従うだけで完了します。

ALS 対応を有効にする（および将来のファームウェア更新を行う）には、以下の手順に従ってください：

1. Asahi カーネルバージョン 6.19 以上を使用していることを確認
2. 使用中のディストリビューションが`iio-sensor-proxy`をデスクトップ環境の依存パッケージとして含んでいることを確認（Fedora Asahi Remix では対応済み）
3. Mac をシャットダウン
4. MacBook の場合は電源ボタンを押し続けて『起動オプションを読み込んでいます』が表示されるまで、またはデスクトップ機の場合は 5 〜 7 秒間押し続ける
5. Apple　のブートメニューから『オプション』を選択
6. macOS　の認証情報でログインし、Command + Shift + T　を押してターミナルを開く
7. `curl https://alx.sh | sh`を実行して　Asahi Installer　を起動
8. プロンプトが表示されたら『Rebuild vendor firmware package』を選択
9. 完了したらインストーラーを終了し再起動

## エネルギー問題を解決

特に Pro/Max/Ultra SoC を搭載したデバイスにおいて、アイドル時の消費電力が長年課題となっています。このプラットフォームの電源管理アーキテクチャは非常に複雑であり、すべてを機能させるためには多くの要素が連携する必要があります。SoC の電源ドメインを管理する Power Manager（PMGR）がありますが、Power Management Processor（PMP）もあって、どういうことですか？

SoC 全体における電源管理の低レベルな詳細はかなり不明瞭です。PMGR は SoC の各領域の電源をオン・オフする役割を担っており、実行時に設定可能です。これはブート時に SoC の一部を電源オンにしたり、サスペンド・レジューム時に電源を制御したりするために必要です。一方、PMP はより興味深い役割を果たしています。多くの SoC ブロックが、共有メモリ領域を通じて自身の電源状態を PMP に直接通知します。また、PMP はアプリケーションプロセッサから、PMGR の電源ドメインがオン・オフされたという報告も受け取っています。PMP がブートされていない場合、これらの報告を読み取らず一部の電源管理機能が動作しません。この情報が具体的に何に使われているのかは明確ではありませんが、おそらくApple Fabric の電源制御やクロッキングの管理などに関わっていると考えられます。

PMP に対応するため、chaos_princess 氏は SoC ブロックと PMGR からの報告を受け取れるようにするドライバを作成しました。これだけで 14インチ M1 Pro MacBook Pro のアイドル時消費電力を約 0.5W 低減できます。これはアイドル消費電力の約 20 %の削減に相当します。もちろん、macOS と同等のアイドル時消費電力やサスペンド時間に到達するためにはまだ多くの作業が必要ですが、これは大きな前進となります。

ベースの M1 SoC を搭載した機器には、古いバージョンの PMP が使用されており、M1 Pro 以降の SoC で見られるバージョンとは互換性がありません。現在、dd-dreams 氏がこれらのマシン向けの PMP 対応に取り組んでいますので、続報をお待ちください。

PMP 対応はまだすべての対応機器で検証が完了していません。上流に統合されるまではデフォルトで有効にして公開する予定はありません。DeviceTree を変更することに慣れているユーザーは、機器の DeviceTree（例：`t8112-j415.dts`）内で `APPLE_USE_PMP` を定義することで、この機能を有効にできます。

## Bluetoothを修正！

Bluetooth と Wi-Fi は長年にわたり干渉や競合の問題を抱えてきました。両者は同じ 2.4GHz 帯の周波数を使用しており、通常は同じデバイス内に共存しています。これを適切に処理しないとパケットロスや音切れが発生します。対処するためのさまざまな手法があります。現代のシステムでは、Wi-Fi と Bluetooth が通常同じコントローラに統合されています。これにより、コントローラはアクティブな 2.4GHz Wi-Fi 接続とBluetooth 接続を同時に監視できます。デフォルトでは各接続に利用可能なエアタイムを公平に割り当てますが、レイテンシが重要な場合や、割り込みを最小限に抑える必要がある接続では問題が生じます。例えば、音声をストリーミング中の接続は、キーボードや 2.4GHz Wi-Fi のスキャンよりも優先される必要があります。

残念ながら Broadcom の場合、物事がいつも単純になるわけではありません。共存設定は Bluetooth Host Controller Interface（HCI）のベンダー固有拡張機能を使用して行われます。これらのコマンドはこれまで上流の Linux カーネルでは対応していなかったため、KDE Connect が Bluetooth スキャンをトリガーした際に Bluetooth コントローラが音声パケットをドロップするなどの問題が発生していました。chaos_princess 氏は最近になってこれらのコマンドをカーネルの Bluetooth スタックに追加しました。また、ユーザースペースの Bluetooth デーモンである BlueZ がすべての音声ストリームを高優先度としてマークしているため、音声をストリーミング中の接続があればいつでもこれらのコマンドを呼び出せます。これにより、Bluetooth 音声のドロップアウトは過去のものとなりました！

## 見えざるところで

ディスプレイコントローラ（DCP）に関するこれまでの格闘はすでに十分に文書化されています。ファームウェアのインターフェースは巨大で、バージョン間で不安定であり、意味不明な制限が数多く存在します。基本的なディスプレイサポートが得られる程度に動作するようになった時点で他のハードウェアに注目が移ってしまい、DCP のさらなる開発に時間を割ける人はいませんでした。しかし、まだ多くの機能が不足しており、特にVariable Refresh Rate（VRR）については頻繁に質問をいただいています。

VRR 対応はかなり複雑です。ディスプレイコントローラとディスプレイは、HDMI と DisplayPort で異なる特殊なパケットを使用して通信する必要があります。また、ディスプレイコントローラは、設定されたタイミングモードの公称リフレッシュレートではなく、実際のフレーム表示に基づいて垂直ブランキング期間を『伸ばす』ように構成しなければなりません。さらに、ディスプレイは VRR を機能として通知する必要があり、これらすべてを KMS API に接続してユーザースペースに通知できるようにする必要があります。

DCP のファームウェアインターフェースは非常に階層的です。他の Apple コプロセッサと同様に RTKit 上で動作しますが、その上位に Endpoint Interface Client（EPIC）が存在します。EPICは『サービス』を公開し、それぞれが関数とコールバックからなる独自の API を提供します。各サービスには『パラメータ』もあり、これはさまざまな実行時動作を制御するキーバリューストアのようなものです。これらのパラメータは、『set parameter』APIを呼び出すことで変更されます。

当初の DCP 初期化作業を行っていた際、外部ディスプレイ用にDCPを電源オンにしたときにあるパラメータが 0 に設定されることが観察されました。これは電源オンシーケンスの一部だと考えられ、Linux ドライバのプローブ処理では電源オンごとにこのパラメータを設定するように記述されていました。

VRR をオン・オフしながらトレースを行っていたところ、同じパラメータが 0x300000 と 0x0 の間で切り替わっていることに気づきました。その後 modeset が行われ、VRR が有効または無効になりました。テストに使用したディスプレイの最小リフレッシュレートは 48Hz です。DCP がリフレッシュレートを表現するために使用する固定小数点形式では 48 は 0x300000 に相当します。もしかして……

Linux の DCP ドライバにコードを追加してこのパラメータを設定し、modeset をトリガーしたところ、期待通りの結果が得られました！

```
RTKit: syslog message: FramebufferDCP.cpp:5931: IOMFBParameter_adaptive_sync Req minRR = 0x300000, mediaTargetRate = 0x0, Fractional Rate = 0
...
RTKit: syslog message: PPipeDCP_H13P.cpp:12251: IOMFB: AdaptiveSync Range [48Hz, 165Hz]
```

これまで設定していたパラメータは、電源シーケンスとは全く関係なく、VRR 用の最小リフレッシュレートを切り替えるためのものでした。modeset の前にこれを 0 に設定すると、単に DCP に対して VRR を無効にするよう指示しているだけでした。通常数百行のドライバコードが必要な処理が、実質2つの関数呼び出しで済むようになったのは、DCP のように巨大なファームウェアブロブを持つ利点の一つです。

ProMotion ディスプレイを搭載した MacBook Pro でも同じ方法で有効になることを確認しました。これは意外な結果でした。MacBook の内蔵ディスプレイは DCP が解析するための EDID/DisplayID を公開していないからです。通常はこれを使ってディスプレイの VRR 機能を読み取り設定します。DCP のファームウェアはまだ完全には理解できていませんが、ラップトップのディスプレイは特別扱いされており、DCP が単に駆動する方法を知っているのではないかと考えられます。これを回避するため、Linux ドライバでは接続されたディスプレイが MacBook Pro の内蔵 LCD であるかを判定し、EDID が存在しない場合でも必要なプロパティを静的に設定するようにしています。

これらを製品レベルに仕上げる作業は少し困難でした。VESA DisplayPort の Adaptive Sync 仕様と KMS API の両方で VRR 状態の遷移に modeset を必要としないことを明確に禁止しています。不幸なことに、 Apple が VESA 仕様を無視しているか、DCP が modeset なしで VRR 状態を遷移させることができないかのどちらかです。そのため、現時点では VRR をユーザースペースに公開できませんし、公開したとしても Kwin などのコンポジタは modeset を必要とするため無視してしまいます。当面は、[プルリクエスト](https://github.com/AsahiLinux/linux/pull/477)がマージされた後、`appledrm.force_vrr`カーネルモジュールパラメータを設定することで、VRR を無条件に強制的に有効にできます。ディスプレイが VRR に対応している場合、DCP は KMS やコンポジタに通知せずに有効にして使用します。Kwin はこの動作を問題なく扱いますが、他のコンポジタでは動作が異なる可能性があります。この挙動に関するバグ報告は、Asahi Linux 側にもコンポジタ開発者側にも行わないようお願いします。コンポジタが体よく許容する必要のないハックです。

ただし、この状況は近い将来変わる可能性があります。HDMI 仕様は VRR 遷移時の modeset を禁止していませんし、他のディスプレイコントローラ（例：Intel 製）も DCP と同様の動作をします。上流ではこの問題をどのように扱うかについて議論が続いています。将来的には、ドライバが VRR モードを無条件に有効にするか、遷移時の modeset を許可するかのいずれかになるでしょう。上流の方針が明確になった時点で、期待通りの方法で VRR を公開し、コンポジタが有効にするタイミングを判断できるようにします。

## 組み込みオーディオのさらなる特性

オーディオスタック全体を上流に取り込む作業は順調に進んでいます。これまでに、Cirrus Logic と Texas Instruments のドライバ（このプラットフォームで使用されるヘッドフォンジャックおよびスピーカーアンプチップ用）、I2S ペリフェラル用ドライバ、Apple DMA コントローラ用ドライバが統合されています。

このプラットフォームの特徴の一つがスピーカーが自己破壊されないようにするための仕組みです。Apple が使用している Texas Instruments のアンプは、接続されたスピーカーの電圧と電流を I2S 経由で SoC に報告できます。これらの値と、接続されたスピーカーの特性[Thiele/Small Parameters](https://en.wikipedia.org/wiki/Thiele/Small_parameters)を組み合わせることで、ボイスコイルの温度をいつでも算出できます。macOS では CoreAudio がこの処理を行っています。Linux では [speakersafetyd](https://github.com/AsahiLinux/speakersafetyd) の役割です。他のデスクトップクラスのプラットフォームでは通常この処理は行われません。まったく行わないか、ハードウェアで完全に処理されるか、[Sound Open Firmware](https://sofproject.org/)によってOSからほぼ隠蔽されるかのいずれかです。

それだけではありません！！各アンプチップのデータ送信ピンは、他のアンプと直列に接続されています。『左』側のスピーカーアンプ群と『右』側のスピーカーアンプ群がそれぞれまとめられ、これらのラインは受信側で 1 つのデータストリームに結合されるため、OR 結合されています。これは、一方の側のアンプが他方の側がデータを送信しようとしているときに論理低レベルであることが保証されている場合にのみ機能します。これは稀なことではなく、Apple が使用しているアンプチップにはこれを支援するためのバスキーパーが搭載されています。バスキーパーはバスを特定の状態に保つための小さな回路で、通常はグランドに接続されます。

当初はこれをアンプチップのドライバ内で完全に設定し、バスキーパー設定を指定するための DeviceTree バインディングを用意していました。しかし、これは上流にとっては制限が大きく柔軟性に欠けていました。そこで、任意の ASoC コンポーネントが構成可能なバスキーパーを持っていることを公開できる汎用的な API を作成しました。これにより、消費側のドライバ（通常はすべての物理オーディオコンポーネントをまとめて1つの『サウンドカード』として扱う『platform』ドライバ）が、実行時に必要なバスキーパー設定を行えるようになりました。バスキーパーは珍しいものではなく、この機能は他の組み込みスタイルのプラットフォームでも有用になる可能性があります。この機能を実装したパッチセットは 7.1 でマージされました。

(訳注:この節の内容は[ユーザス空間音声スタック](https://asfdrwe.github.io/asahi-linux-translations/sw/audio-userspace/)が参考になります)

## macOSを超えて

完全に文書化されていないプラットフォームを立ち上げる際の課題の一つは、ハードウェアの能力についての基準がまったくないことです。現時点で判明している限り、ハードウェアは macOS が行う処理と同等のことしかできません。m1n1 および Linux ドライバのコードのほとんどは、macOS が特定のハードウェアブロックに送信するコマンドを観察し、それを再現する形で作成されています。でも、*常に*そうであるとは限りません。

Apple は購買力を活かして、サードパーティのチップベンダーから自社が求めるものを確実に手に入れることを厭いません。基板レベルの修理や Right to Repair 関連のインターネット上の人物をフォローしている方々は、Apple が Texas Instruments から CD321x シリーズのチップをどのように入手しているかを知っているでしょう。これらは公開データシートが存在しない TI の USB-C mux チップの互換性のない変種であり、Apple に対してのみ大量に販売されています。スピーカーアンプチップについても同様の扱いで、TI の TAS2764 および TAS2770 の変種です。CS42L84 も、Cirrus Logic の CS42L42 の Apple 専用変種です。ベンダーはこれらの変種の存在すら認めないことが多いです。公開されている同等品との関連性は、リーク情報やインターネット上の調査によってのみ推測できます。

これらの変種は公開されている同等品と十分に似ているため、データシートやドライバの情報が有用な参考資料になります。非互換性は通常レジスタレイアウトのわずかな違いや他の部分に影響しない個別の機能に限定されます。TI のスピーカーアンプの場合、Apple 変種対応を上流の TAS2764 および TAS2770 ドライバへの最小限の変更で追加できました。

とはいえ、まずは macOS のトレースから始めて、これを唯一の真実の情報源として扱うのが常に有益です。CS42L84 は CS42L42 とは十分異なっていたので、独自の Linux ドライバが必要となり、すでに上流に取り込まれました。ただし、このアプローチには macOS が使用している機能しか公開できないという制限があります。macOS が CS42L84 を 48kHz または 96kHz でのみ動作させるため、Linux ドライバでも当初はこの 2 つのサンプリングレートしか対応できませんでした。これは PipeWire が 48kHz または 96kHz 以外の音声ストリームをリサンプリングするために CPU サイクル（およびバッテリー）を浪費させることになり、音質も低下させるという大きな制限でした。しかし、CS42L42 は他の一般的なサンプリングレートすべてに対応しています。レジスタレイアウトとプログラミング手順は異なりますが、48kHz と 96kHz で実際に書き込まれる値は両チップで同じです。そこで、CS42L42 のデータシートから他のサンプリングレートの値をそのまま取得して CS42L84 ドライバに追加してみたらどうなるでしょうか？結果、それらのサンプリングレートも対応するようになりました！

ヘッドフォンジャックの入出力両方で 44.1kHz、88.2kHz、176.4kHz、192kHz のサンプリングレートにハードウェアで対応するパッチは、上流に直接提出され、7.1 でマージされました。また、この変更を Asahi カーネル 6.19.9 にバックポートしたため、ユーザーはすぐにこの機能を利用できます。

## M3のさらなる対応！

M3 搭載マシン向けのさらなるハードウェアサポートパッチも Asahi カーネルツリーに取り込まれています。これには PCIe、MacBook のキーボードとトラックパッド、SMC ベースの RTC およびリブートコントローラ、NVMe コントローラのサポートが含まれます。これらは Michael Reeves 氏と Alyssa Milburn 氏による再度の貢献です。これにより、M3 に対する Linux 対応は、M1 向けの最初の Asahi Linux アルファ版とほぼ同等のレベルになりました。

Asahi Installer 経由での M3 機器へのインストールをまだ有効化する準備はできていませんが、着実に進捗しています。続報をお待ちください！

## Fedora Asahi Remix 44

Fedora Asahi Remix 43　のリリースが遅れてしまったにもかかわらず、Fedora Linux 44　のリリース日である 4 月 28 日またはその数日以内に、Fedora Asahi Remix 44 をリリースする予定です。今後は、新規の KDE Plasma ベースインストールで、Plasma 6.6 に搭載されたいくつかの新機能を利用できます。

[Plasma Setup](https://kde.org/announcements/plasma/6/6.6.0/#plasma-setup) が従来の Calamares ベースのセットアップウィザードに代わり、ユーザーアカウント作成とシステム設定に Plasma ネイティブの体験を提供します。また、Plasma Login Manager がデフォルトの greeter およびセッションマネージャとなり、SDDM に置き換わります。これは新規インストールのみに適用され、以前のバージョンの Fedora Asahi Remix からアップグレードしたユーザーの設定は変更されません。

Fedora Asahi Remix 44では、ベンダー提供の Mesa および virglrenderer パッケージも廃止されます。まだ手動で移行していないユーザーは、Fedora 公式リポジトリから提供される上流の Mesa およびvirglrenderer パッケージに自動的に移行されます。

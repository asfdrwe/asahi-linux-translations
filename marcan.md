# Asahi Linux のプロジェクトリーダーを辞任

2000年代後半、私は Wii の homebrew(訳注:自作ソフトを動かすこと) シーンに大きく貢献しました。当時、ニンテンドー Wii 上でユーザーが自分の非公式アプリを実行できるようにするためのソフトウェア（最近では『脱獄(jailbreak)』と呼ばれている）に取り組んでいました。

自分の仕事と所属していたチーム（Team Twiizers、後のfail0verflow）に情熱を持っていました。にもかかわらず、燃え尽きてしまいました。主に自分が特別だと思っているユーザー(entitled user)の割合が非常に大きかったからです。私たちのソフトウェアを使用しているほとんどの人々は、海賊版のゲームをプレイしたいだけでした（海賊版に対応せず、容認もせず、直接許可もしませんでした）。プラットフォームをオープンに保つためにメーカーといたちごっこを続けていましたが、私たちの努力は、ただ他人の作品を盗みたいだけで、大声で権利を主張する人々によって利用されるのを目の当たりにしただけでした。しばらくすると、本当にうんざりしてきました。新しいゲーム機がリリースされるにつれ、私は純粋に楽しむためにLinuxの移植に集中するようになり、コミュニティを作ろうとせず、結局海賊が使う道具になる脱獄や脆弱性に取り組もうともしませんでした。

アップルが M1 をリリースしたとき、Linuxを走らせることが私の夢のプロジェクトだと気づきました。技術的な課題は、過去のコンソール homebrew プロジェクトと同じでしたが（実際はもっと大きかったですが）、今回はプラットフォームがすでにオープンだったため、脱獄は必要なく、ソフトウェアを海賊版にしたい揉めごとや自分が特別だと思っているユーザーを心配する必要もありませんでした。そして、M 1上で Linux を動かすことは、PS4 上で動かすよりもずっと大きな仕事でした。

私は Asahi Linux プロジェクトを立ち上げ、膨大な量のサポートと寄付を受けました。信じられないことに、プロジェクト立ち上げの呼びかけからわずか数日後に、プロジェクトを実現するために必要なサポートを得ることができ、仕事に取り掛かりました。最初の2、3年は素晴らしく、何もないところから、ラップトップで得られるLinux体験の中で最もスムーズなもののひとつにまでプラットフォームを引き上げました。もちろん、ハードウェア対応がまだ不足している部分もありましたが、全体的なエクスペリエンスは、ほとんどのx86ラップトップで得られるものに匹敵するか、それ以上でした。そして、ベンダーのサポートや文書を一切持たずに、すべてをゼロから作り上げました。ありえない偉業であり、これまでになかったことでしたが、私たちはそれを成し遂げました。

残念なことに、しばらくすると面白くなくなっていきました。まず、Linuxカーネルへの上流コードの問題がありましたが、これについてはすでに長々と話したので、ここでは繰り返しません。実質的にすべてのLinuxサブシステムにまたがるコードを上流に統合しなければならない立場にあり、あらゆるカテゴリーのドライバやいくつかの共通コードに触れなければならないことは、信じられないほどフラストレーションのたまる経験でした。

しかしその後、自分が特別だと思っているユーザーも現れました。今回は、ゲームを盗むことではなく、機能についてでした。「サンダーボルトはいつ来ますか？」「Asahi は USB-C 経由でモニターを使えるようになるまで役に立たないです」「macOSと比べてバッテリーの持ちは最悪です」(x86ラップトップと比べたら誰も文句は言わないのですが…)）「CPUの温度もチェックできないです」 （そう、これは実際に言われました）

そしてもちろん、「M3/M4対応はいつ来ますか？」

長い間、安定版リリース後も、人々は Asahi Linux、特に Fedora Asahi Remix を「アルファ版」「不安定」「日常の運用には適さない」と主張し続けました（私を含め[何千人ものユーザ](https://stats.asahilinux.org/)が日常的に使用し、サーバにも使用しているにもかかわらず）。

私たちがどれだけやっても、どれだけありえない偉業を成し遂げても、人々は常にそれ以上を求めました。さらにもっと。その一方で、プロジェクト開始以来寄付金や誓約金は徐々に減り続けています。短期的には、フルタイムで『Asahi』に携わるという私の夢がすぐに破滅を迎えるほどではないですが、本当に感謝されているのだろうかと考えさせられるには十分でした。毎月の寄付額がピークに達したのは、最初の1、2ヶ月でした。私たちが成し遂げれば成し遂げるほど、支援は減っていくように思えました。

私は燃え尽き症候群が現実的なリスクであることを知っていたので、カーネル上流など特定の分野に費やす時間を制限することで管理しました。これはそれなりにうまくいき、その頃はほぼ持続可能でした。

そして2024年になりました。詳しくは書きませんが、昨年は個人的な事情により、信じられないほど波乱に満ちた年でした。結局、私は1年の大半を家から離れて過ごすことになり、その間、私や家族に嫌がらせをしたり攻撃したりするさまざまな虐待者やストーカーに対処しなければなりませんでした（そして今も続いています）。

2024年でも少し前進しましたが、これは自身を非常に弱い立場に置いたままでした。Asahi の仕事は思ったほどできていませんでしたし、ユーザーはより多くの機能と機器対応を要求するようになりました。

規格準拠 Vulkan ドライバと、x86-64のゲームやアプリのためのエミュレーションスタック全体を公開しましたが、DP Alt Mode　がないまま立ち往生してしまいました（この機能を実現するには、深いリバースエンジニアリングやデバッグやカーネルの手術が必要で、適切かつ堅牢に実装するには、特定のカーネル・サブシステムを大幅にリファクタリングするか、あるいはまったく新しいサブシステムを導入する必要があります）。

今年の初めから徐々に仕事を再開し始め、前年はほとんど仕事ができなかったことに強いストレスと罪悪感を感じていました。DP Alt の 　『完全』対応はまだ先ですが、最初の1、2ヶ月は、各マシンの種類ごとに、特定のタイプ C ポートでのみ動作する限定バージョンを公開したいと思っていました。12月に Sven 氏が PHY のコードを少し進めたので、私はそれを拾い上げ、3つのドライバのコードをほぼ確実に動作するような形に叩き込みました。最善のアプローチではなかったとはいえ、カーネルコミュニティとまた大規模な流域議論をすることなく、なんとかできる最大限のことでした（メーリングリストでもこの話題を[持ち出そうとした](https://lore.kernel.org/lkml/fda8b831-1ffc-4087-8e7b-d97779b3ecc5@marcan.st/T/#u)が、あまり反応はなかったです）。

Rust for Linux が上流Linuxプロジェクトとして生き残るために抱えてきた問題は、よく文書化されているので、ここでは詳しく繰り返しません。私は、LinuxへのRustの統合に関するLinus氏の対応は、リーダーシップの大きな失敗だと考えています。このような大きなプロジェクトが生き残るためには、主要なステークホルダーからの大きなサポートが必要です。一方、Linus氏の下流にいる複数のサブシステムメンテナは、プロジェクトを妨害したり、妨げたり、容認できない暴言を吐いたり、一般的に士気を傷つけたりすることに全力を尽くしていて、何の結果ももたらしていません。Rust for Linux の主要なメンテナの一人は、すでに数ヶ月前に辞職しています(訳注:[この記事](https://xenospectrum.com/conflict-over-the-introduction-of-rust-intensifies/)など参照)。

ご存知の通り、私たちは Asahi のために Rust for Linux に賭けてきました。単なる遊び（あるいは単なるメモリ安全性）でもありません。Rust は、GPUドライバが時間内に成功できた理由のすべてです。現在、下流ツリーにはさらに2つの Rust ドライバがあり、3つ目のドライバは C から Rust に書き換えられる予定です。なぜなら、Rust の方が私たちが直面する固有の課題にはるかに適しており、C　ドライバはメンテナンス不能になりつつあるからです。ちなみにこれは、Nvidia GPU 用の新しい Nova ドライバが Rust で書かれているのと同じ理由です。当然のことながら、より複雑で斬新な課題を持つ最新のハードウェア用のドライバを書くには、よりモダンなプログラミング言語の方が適しています。

なぜもっと長い期間、おそらくあと数年、Rust の状況を放っておいて、それまで下流で物事を維持しつづけることができないのかと思う人もいるかもしれません。理由のひとつは、もちろん、この状況が現在の開発者の士気を傷つけているからです。もうひとつは、私たちの　Apple GPU　ドライバ自体が、Linux　向け　Rust　が目的に適っているという大きな証拠だからです（Rust　でゼロから書かれた最初の大きなドライバであり、Rust　カーネル抽象化の多くの開発をもたらしました）。上流を目指さないことは、単に関心がないとみなされ、Rust for Linux の取り組みの存続の可能性を損なうかもしれません。しかし、それだけではありません。

実際、Linux カーネルの開発モデルは（おそらく逆説的に）上流を奨励し、下流でのフォークを罰するように設計されています。上流に関心を持たず、完全なハードフォークを維持することは可能ですが、これは長期的に実行可能な解決策ではありません(こうやっているから2年で枯れてしまうベンダーのアンドロイドカーネルツリーが発生します）。Asahi Linuxの下流ツリーは継続的に最新の上流カーネルの上にリベースされています。しかし、それだけではありません： カーネル/Mesa ポリシーでは、GPU ドライバの上流 Mesa 対応は、カーネル側が統合準備できるまで、統合して有効にすることはできません。これは、私たちも Mesa フォークをユーザーに提供しなければならないことを意味します。私たちの GPU ドライバは99% Mesa の上流に統合されていますが、意図的にハード無効化されており、カーネル側が対応するまで、有効にするような変更を提出することは許されていません。これは実際には、ユーザーが GPU アクセラレーションをコンテナ技術（Docker/Podmanのようなものだけでなく、Waydroidのようなものも含む）とともに動作させることができないことを意味します。Flatpakでは[部分的な回避策](https://pagure.io/fedora-asahi/mesa-asahi-flatpak)はありますが、他のすべてのコンテナシステムは機会がありません。このように、Linuxカーネルの上流への統合が困難であることが、今日の下流ユーザーに打撃を与えています。

私は不正を見たら放っておくタイプではないので、別の長期メンテナがその立場を悪用して R4L (訳注: Rust for Linux)を妨害し、上流の進展を妨げようとしたとき、私は声を上げました。そして(かなり広く取り上げられている）その対応が、私を崖っぷちに追い込む最後の一滴となりました。Apple ARM対応の上流メンテナーの立場を辞しました。そのスレッドの後半で、別の主要なメンテナが[「我々がThin blue lineだ」](https://lore.kernel.org/lkml/20250208204416.GL1130956@mit.edu/)(訳注:Thin blue lineは[警察官](https://en.wikipedia.org/wiki/Thin_blue_line)という意味)と皮肉たっぷりに発言しましたが、誰も気にしていませんでした。この人物は、以前Rust for Linuxのメンテナを辞めさせた[人物](https://lore.kernel.org/lkml/20240828211117.9422-1-wedsonaf@gmail.com/)と同一です。

しかし、それは公の事件にはとどまりません。その後の数日間で、カーネルとそれに隣接する Linux スペースの何人かのメンバーが、私やAsahi Linux を支持しているように見せかけながら、密かに私を恨み、密室で恨みを募らせるという、二枚舌の駆け引きを私としてきたことを知りました。このようなことは、誰も私に個人的な電子メールを送ったり、何が起こっているのかを教えてくれたりすることなく起こりました。そのうちの一人は、Asahi Linux が生き残るために交流しなければならない複数のプロジェクトで高い地位にある人物で、私を直接罵倒し嫌がらせをした人物の味方をし、今もその味方を続けていると聞きました。どうやら、私が Asahi で働くために誰かに雇われているというような、暗示的な虚偽もあったようです（違います、ホスティングのために[bunny.net](https://bunny.net/)が無料のCDNクレジットを与えてくれている以外、企業スポンサーはゼロです）。

私のマストドン投稿が気に入らない人がいることは承知しています。たしかに、私はときどき険悪になることがあるし、それは私自身の責任でもあります。しかし、これは許されることではありません。裏で徒党を組んだり、自分の意図に嘘をついたりする人たちとは一緒に働けません。コミュニティで本当に有害な人たちではなく、メッセンジャーに責任を負わせる人たちとは一緒に働けません。公の場でのコメントに腹を立て、私的な場では何も変わらないように見えるにもかかわらず、私的な場で物事を処理した方がいいと主張する人たちとは一緒に働けません。何千人ものフォロワーに対してソーシャルメディアで不品行を呼びかけることを非難する一方で、自分自身はソーシャルメディアと何千人もの購読者を持つメーリングリストの両方で人々を非難する人々とは一緒に働けません。公の場で政治的・差別的な言葉を使い、何の影響も受けないような高い地位にいる人たちとは一緒に働けません。主要なサポーターやメンテナが積極的に辞職し、Linuxカーネルには10フィートポールで触れないというメッセージをあらゆる人々から受け取り続けているのに、私の方が問題で、すべてがうまくいっていると言う人々とは一緒に働けません。

Apple が M1 をリリースしたとき、Linus Torvalds氏は M1 で[Linuxが動くことを望みました](https://thenextweb.com/news/linus-torvalds-wants-apples-new-m1-powered-macs-to-run-linux)、実現の望みは薄かったです。私たちはそれを実現し、[Asahi Linuxが動作するM2 MacBook Air](https://lore.kernel.org/lkml/CAHk-=wgrz5BBk=rCz7W28Fj_o02s0Xi0OEQ3H1uQgOdFvHgx0w@mail.gmail.com/T/#u)からLinux 5.19がリリースされました。私は、Linus氏の熱意が私たちのコミュニティへのサポートにつながり、上流での苦労を助けてくれることを期待していました。悲しいことに、それは実現しませんでした。2023年11月、私はLinus氏に、カーネルへのコントリビュートとメインテナンスの課題について話し合い、私たちがどのように手助けできるかを確認するための[招待状](https://gist.github.com/marcan/fe70ee6648f3d5ae94eb8332265b8d95)を送りましたが、返事はありませんでした。

2011年、Con Kolivas氏が[Linuxカーネルコミュニティを去りました](https://web.archive.org/web/20110707151924/http://apcmag.com/why_i_quit_kernel_developer_con_kolivas.htm)。昼間は麻酔医だったCon Kolivas氏は、間違いなく最後の偉大なLinux カーネルの趣味のハッカーでした。それ以来、状況は悪化の一途をたどっています。今日、Linux の重要なメンテナーやクロスサブシステム・コントリビューターとして生き残ることは、企業に雇われていなければ事実上不可能です。Linux は趣味のプロジェクトとして始まりましたが、その趣味のルーツはすっかり失われてしまいました。

Asahi Linuxを始めたとき、私は生活のほとんどをLinuxに支配されてしまいました。趣味のほとんどをあきらめ（結局、Asahi Linuxが私の夢の趣味でした）、フルタイム以上の時間をプロジェクトに費やしました。当時は楽しかったですが、今はもう楽しくないです。M3 Pro が箱に入っていて、まだ電源すら入れていません。立ち上げ作業をするのが怖いのです。手間をかける価値があるとは思えません。

まだ公開していない機能のことを気にせず、リラックスできる自由な時間が恋しい。[音楽を作りたいです](https://youtube.com/@TsuiokuCircuit)。ジャムセッションが恋しい。友人や家族とディナーに出かけて、まだどれだけ上流に統合していないかを気にしなくていいのが恋しい。罪悪感を感じることなく、座ってゲームをしたり、映画を見たりできることが恋しい。

私はただちに Asahi Linux プロジェクトのリーダーを辞任します。プロジェクトは私なしで継続します。現在責任と管理者資格の移譲を処理するためにチームの他のメンバーと協力しています。個人的なPatreonは一時停止され、私個人を支援してくださった方は、[Asahi Linux OpenCollective](https://opencollective.com/asahilinux) に支援を移行してください（GitHub Sponsors では、私が一方的に支払いを一時停止することはできませんが、スポンサーにはこの変更が通知されますので、手動でスポンサーをキャンセルすることができます）。

Asahi Linux のチーム全員に感謝したいです。彼らがいなければ、私は一人ではどこにも行けなかったでしょう。みんな自身が何者か知っています。また、このプロジェクトを実現可能なものにしてくれたPatreon とGitHubのスポンサーの皆さんにも最大限の感謝を捧げます。

私を雇うことに興味がある方、またはそのような方をご存知の方は、ぜひご連絡ください。リモート・ポジションのみ、または、コンサルティングまたはフレキシブル・タイム／非専属ベースでお願いします。連絡先：[marcan@marcan.st](mailto:marcan@marcan.st)。

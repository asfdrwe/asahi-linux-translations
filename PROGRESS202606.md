# 進捗報告：Linux 7.1

- [前回](https://github.com/asfdrwe/asahi-linux-translations/blob/main/PROGRESS202604.md)

Linux 7.1 がリリースされました。進捗報告をお届けします。今回は **M3** の進捗、Apple のバグ修正などなどお伝えしますよ！

### Master Boot Record の復活

Mac で電源ボタンを長押ししてブートピッカーを表示させる（または Startup Disk アプリケーションを使用する）ときに、『Asahi』として表示される項目は、実際にはオペレーティングシステムが入っているパーティションではありません。Apple のブートツールは、APFS コンテナ内に『有効な』macOS インストールがあると認識した場合にのみ動作します。

Asahi Installer が Apple のブートローダーを使用し、ユーザーが Recovery から毎回コマンドを実行しなくても済むようにするため、Asahi Installer は小さな APFS コンテナ（2.5 GB）を作成し、m1n1 をカーネルとする macOS インストールであると Apple のツールに認識させるのに十分な macOS の一部を入れています。この構成は macOS 12 から macOS 26 まで完全に変更なく動作していました。Apple はまた、実際の XNU カーネルではない raw バイナリをブートしようとしたときにのみ発生するバグをいくつか修正しました。

しかし、macOS 27 Golden Gate 開発者ベータ版がリリースされた直後、Linux にブートできなくなったという報告が相次ぎました。Startup Disk とブートピッカーから Asahi の選択肢が完全に消えてしまったのです。これは非常に深刻な問題だったので、調査を優先的に進めました。

diskutil でディスクを調べたところ、macOS 27 にアップグレードしても Asahi 関連のパーティションはすべて残っていました。データ消失は発生しておらず、これは良い兆候でした。また、同一マシン上で macOS 26 のブートツールを使用すれば Asahi は依然としてブート可能でした。

chaos_princess 氏が Apple 公式の macOS Installer や、以前 Apple のブートツールを調べていた古い記録を調査しました。macOS Installer は再起動前に APFS のメタデータを設定しており、それがボリュームを『ブート可能』とマークするフラグであることがわかりました。macOS 27 まではブートツールはこのフラグを完全に無視していましたが、Asahi の APFS コンテナにこのフラグを手動で設定すると、他の修正なしに macOS 27 のブートピッカーに表示されるようになりました。

今後、すべての新規 Asahi インストールでは Asahi Installer がこのフラグを自動的に設定します。また、既存のインストールを修正するインストーラモードも追加しました。macOS 27 開発者ベータ版をインストールして Asahi にアクセスできなくなった場合は、インストーラを再度実行し、『Fix macOS 27 boot picker compatibility』オプションを選択してください。

chaos_princess 氏は Linux から実行できる修正プログラムも開発しました。将来的にはこの修正を自動適用したいと考えていますが、信頼性が高く、誰かのファイルシステムを破壊しないことを確認するためのテストデータがまだ必要です。そこで皆様のご協力をお願いします。

テストにご協力いただける方は、[このリポジトリ](https://github.com/AsahiLinux/asahi-fix27) を clone し、**macOS 27 にアップグレードする前に** Linux 上でビルド・実行してください。Asahi ボリュームが macOS からブート対象として選択可能になっていれば成功です。何か問題があれば、OFTC または Matrix の[チャンネル](https://asahilinux.org/community) までお知らせください。

### 3 バイトが引き起こすシャットダウン

macOS 27 は、グローバルファームウェアを持つすべての周辺機器（SMC を含む）のファームウェア更新も行います。SMC の数多くの機能の一つにバッテリー管理があります。Linux 電源供給ドライバは、SMC と通信して充電状態、電圧、残り時間、バッテリーの健康状態などの情報を取得しています。また、SMC のファームウェアインターフェースを使って充電開始・停止の閾値を設定し、バッテリーの寿命を延ばしています。macOS 27 の SMC ファームウェアでは、バッテリー管理インターフェースの一つが 32 ビット整数を返すものから 1 バイトを返すものに変更されました。これによりドライバが混乱し、特定の条件下でバッテリーが故障したと判断してシステムを保護するための緊急シャットダウンを開始してしまいます。この問題はすでに下流のカーネルで修正済みです。バージョン 7.0.12 以降の電源供給ドライバは、両方のファームウェア ABI に対応しています。

### ベータ版のインストール

このようなバグは、開発者ベータ版はあくまで開発者向けであることを改めて思い起こさせるものです。本番環境のマシンにインストールすることはおすすめしません。これまで発生した 2 つの問題は幸い軽微でしたが、今後もすべてがそうとは限りません。グローバルファームウェア更新は実質的に永久的で、DFU リストアでしかロールバックできません。今後は開発者ベータ版のインストールを控えてください。私たちはテスト用の犠牲マシンでこれらを検証しています。高価なハードウェアと大切なデータを危険にさらす必要はありません。

### 変わらぬもの…

コンピュータプラットフォームやその中に搭載される IC の設計・検証は非常に高価で時間のかかる作業です。必要のない変更を既存の設計に加える意味はほとんどありません。プロジェクト初期に、Apple もそれに同意し、不要な破壊的変更を加えないだろうという賭けをしました。GPU のような大規模 SoC ブロックを除き、この賭けはほぼ成功しています。

Apple Silicon ラップトップのオーディオは、いくつかの異なる IC と SoC ブロックで構成されています。オーディオ IC の事実上の業界標準は I²S（I²C ベースのオーディオ最適化バス）です。Apple の I²S コントローラは M1 以来変更されていません。これらのオーディオ IC はすべて安定したクロックソースを必要とし、多様なオーディオデータレートに対応できるように構成可能でなければなりません。Apple の数値制御発振器（Numerically Controlled Oscillator, NCO）も M1 以来変更されていません。また、Apple Silicon マシンのほぼすべてで同一のスピーカー・アンプチップとヘッドセットアンプチップが使用されています。そのため、chaos_princess が M3 機器向けにスピーカーとヘッドフォンジャック対応を追加した際には、ささいな DeviceTree の追加と asahi-audio および speakersafetyd の設定ファイル程度で済みました。これにより、**M3 マシンでも Asahi Linux で高品質なオーディオ出力が可能になりました**！

M3 マシンでは、CPU 周波数切り替えと適切な big.LITTLE タスクスケジューリングのサポートも追加されました。Apple は M2 ベース以降、CPU 周波数切り替えの仕組みを変更していません。そのため、M3 および M3 Pro/Max/Ultra SoC は既存の cpufreq ドライバに対して Device Tree の変更だけで動作します。これにより、タスクは必要に応じて効率コアまたはパフォーマンスコアに賢く配置され、CPU コア自体も負荷に応じてクロックを上げ下げします。これにより**省電力と性能向上の両方が実現**されます！

SMC のハードウェアセンサーサポートの追加も同様に簡単でした。SMC のファームウェアはマシン間で大きく変わらないため、これも DeviceTree の変更だけで済みました。

その他、PCIe、WiFi、Bluetooth、NVMe、キーボード、トラックパッド、その他の主要 SoC ブロックドライバも M3 シリーズで Linux 上で動作しています。この作業の多くは [Yureka 氏](https://fedi.yuka.dev/yuka) によるものです。Yureka 氏は M3 シリーズマシンを使って m1n1 と Linux の両方で精力的に作業を続けています。Asahi Installer での正式サポート開始まではまだ道のりがありますが、進捗は非常に速いです。続報をお楽しみに！

### ファームウェアを自前で書く時代に？

このプラットフォーム上の複雑なハードウェアのほとんどは、複雑なファームウェアブロブを使用しています。多くは RTKit（Apple がカーネルとハードウェアの間に標準化されたインターフェースを提供するために使用する RTOS 的ファームウェアフレームワーク）に基づいています。ただし例外もあります。DCP や AOP のように RTKit を基盤にしつつ、さらに EPIC という抽象化レイヤーを重ねているものや、Broadcom の WiFi/Bluetooth チップセットのように Apple が直接制御していないサードパーティファームウェアを使用しているものがあります。そして **Apple Video Decoder (AVD)** です。

AVD は特別です。そのファームウェアは RTKit でも EPIC でもなく、第3の秘密です。ハードウェア自体は基本的に ARM Cortex-M3(訳注:以後 CM3) が、AVC（H.264）、HEVC（H.265）、VP9、そして最近の SoC では AV1 などの動画をデコードするための固定機能ハードウェアユニットを制御しています。CM3 はファームウェアブロブを実行し、XNU が動画データを渡すためのインターフェースを公開し、デコーダハードウェア自体をプログラミングします。通常なら問題ありませんが、Apple は AVD のファームウェアと大量の設定データを **AVD kext の中にバンドル**するという興味深い選択をしました。さらに、各 SoC で AVD のバリエーションがわずかに異なります。これにより、Asahi Installer が Apple の kext 内ファームウェアオフセットの変更に常に追従しなければならず、非常に煩雑になります。**では、自前のファームウェアを使ったらどうでしょう？**

XNU がロードするファームウェアは CM3 によって検証されません。シグナルを受け取ると、実際に何が入っているかに関係なく、リセットベクターから実行を開始します。では、ただ…私たち自身のファームウェアを使ったらどうでしょう？

ファームウェアの役割は下層の動画デコーダハードウェアを抽象化することだけなので、さまざまなハードウェアブロックの割り込みハンドラをインストールするだけで十分です。下層ハードウェアの動作を理解していれば、Linux ドライバから直接プログラミングできます。

AVD ファームウェアは標準的な Cortex-M3 コードなので、エミュレータ（QEMU など）で実行可能です。Jamie 氏と R 氏と Eileen 氏が過去に AVC と VP9 デコーダに必要な命令とデータ形式をリバースエンジニアリングしてくれました。

XNU kext は各 AVD リビジョンごとに独自のチューナブルを適用します。これらを Linux カーネルドライバで上流に維持するのは困難なので、ファームウェア側で扱うのが適切です。

長らく進展が少なかったのですが、新規貢献者の [sofus 氏](https://github.com/sofus13) が取り組み、割り込みハンドラのインストールと各バリアントのチューナブル適用を行うカスタム AVD ファームウェアを作成しました。これにより **AVC ハードウェア向けの動作する V4L2 ドライバ** が完成しました！このハードウェアは 10-bit AVC 動画を最大 4K までデコード可能で、V4L2 Request API を実装したソフトウェアで良好に動作します。ファームウェアをシンプルでステートレスに保ち、ユーザースペースとカーネルが動画データの解析とデコーダのプログラミングを担当することで、将来的に VA-API や Vulkan Video などの他の動画アクセラレーション API への対応も容易になります。

ユーザーに AVD 対応を提供できるようになるまでには、まだいくつかの作業が残っています。AVD は VP9、HEVC、そして一部の SoC では AV1 さえも対応していますが、これらのいずれについてもまだ対応を実装していません。一部のデバイスにはクセがあり、テストしてドライバで適切に扱う必要があります。それほど遠くない将来に、みなさんに提供可能な状態のものを用意できると願っています！

### 大規模な m1n1 リリース

最近、m1n1 のバージョン 1.6.0 をタグ付けしました。これはディストリビューションにとって重要なリリースです。なぜなら、stage 2 のビルドに Rust を必要とする初めてのバージョンだからです。以前は、m1n1 は chainloading 対応付きでビルドする場合にのみ Rust を使用していました。Stage 1 m1n1 は Apple のブートツール内の XNU カーネルを置き換えるもので、EFI System Partition をマウントし、そこから Stage 2 m1n1 を chainload するためだけに使用されます。しかし少し前、GPU 初期化を m1n1 に移動させるという決定を下しました。これにより、カーネルドライバが Apple のハードウェア初期化データに含まれる浮動小数点数を扱う必要がなくなり、DeviceTree バインディングも大幅に簡素化されました。そのため、将来的に Linux Kernel Mailing List に送信する GPU ドライバのバージョンは、この初期化を m1n1 に依存することになります。また、Apple DeviceTree の解析コードを Rust に移植しました。これは m1n1 のほぼすべての他の部分で使用されています。

m1n1 はファームウェアであるため `no_std` Rust を使用し、`aarch64-none-softfloat` をターゲットとしています。余計な依存を避けるため、`make` に `BUILDSTD=1` を渡すと `core` と `alloc` をフル softfloat ツールチェーンなしでビルドできます。

バージョン 1.6.0 では M3 シリーズ対応の大幅な改善（SPMI コントローラと PCIe 初期化のサポートなど）も行われました。また、[kisd](https://github.com/AsahiLinux/kisd) を使用して SoC のハードウェア UART を DebugUSB 経由でトンネリングできるようになり、Central Scrutiniser と同等の機能が実現できます。この作業の多くも Yureka 氏によるものです。

さらに、M4 および A18 Pro（MacBook Neo）サポートの基盤整備も進めており、Apple の非 macOS ブートモードの改善や Apple DeviceTree の新しい電源ドメインメタデータ対応も行っています。
